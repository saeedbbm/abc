# Follow-Up Plan (Pipeline + UI)

## Anti-Goals And Non-Regression Rules
- Do NOT do a big-bang rewrite. Make changes in narrow phases, keep current working behavior, and only move to the next phase when the current one is stable.
- Preserve the useful work already completed: demo-state baseline/workspace/reset flow, run/demo scoping, and other working latest-run fixes. New work should build on top of that, not destabilize it.
- Keep current API/UI contracts stable while migrating. Add new fields, backfills, and QA artifacts before removing old inference paths.
- After every accepted phase, verify these surfaces still load and still use the latest scoped data: `Discover`, `Tickets`, `Guides`, `Chat`, `KB`, `Admin Demo State`.
- If a phase fixes one demo surface but regresses another already-working surface, stop and fix the regression before moving forward.
- Keep the no-cheating boundary explicit at every phase: `ground-truth/*` and `goals/*` may be used for evaluation, scoring, and rerun decisions only; never leak them into runtime prompts or runtime materialization logic.

## Shared Iteration Loop For Every Phase
- 1. Capture the current baseline: latest run ID, active demo state, and the current outputs on all demo surfaces.
- 2. Make the smallest code/data-flow change that should improve the active phase only.
- 3. Rerun from the earliest changed step, not the whole pipeline unless required.
- 4. Re-score the result using:
- `goals/goal.txt`
- `demo_story.txt`
- `ground-truth/A-projects.txt`
- `ground-truth/B-decisions.txt`
- `ground-truth/D-team-members.txt`
- `ground-truth/E-customer-feedback.txt`
- `ground-truth/F-entity-counts.txt`
- `ground-truth/G-key-pages.txt`
- 5. Inspect the live product surfaces, not only the step artifacts.
- 6. Compare the new result to the previous accepted result and ask: did this improve the target phase, and did it avoid regression elsewhere?
- 7. If improved and non-regressing, keep it and advance. If not, revise and rerun the SAME phase again.
- 8. Do not move to the next phase until the current phase exit gate passes.

## Content Language Quality Gate (applies to every phase that generates text)

Every phase that touches prompts, generates text, or materializes user-facing content must pass a language quality gate before advancing.

### The rerun-until-readable loop
- 1. After every prompt change or rerun, sample 5 generated outputs of each type affected (ticket titles, ticket descriptions, verify card titles, entity page items, how-to sections, human page paragraphs).
- 2. Read them as a user would. Ask: would I show this to a non-technical person without embarrassment?
- 3. Fail the gate if ANY of these are true:
  - A title is longer than 10 words.
  - A description starts with "A Jira ticket for..." or "This entity represents..." or "This project involves..."
  - A title contains "may", "potentially", "appears to", "based on available evidence", "it seems that".
  - A verify card title tries to encode both the diagnosis and the uncertainty in one sentence.
  - A how-to section is generic advice that does not reference any specific team convention, person, or source.
  - An entity page item is a paragraph crammed into a bullet point (more than 2 sentences).
  - Any text uses jargon that is not the actual technical term from the source material.
- 4. If the gate fails: rewrite the prompt, rerun the step, re-sample. Do NOT advance.
- 5. If the gate passes: record the passing sample and move on.

### What good language looks like
- Titles: short, scannable, entity-first. "Kim Color Convention — Missing Source", not "The color convention attributed to Kim may be missing a source reference from the 2024 conversation".
- Descriptions: what to do, in 2-4 sentences. Not a restatement of the title with more hedging.
- Bullet items: one fact per bullet. Active voice, short sentence.
- No filler intros ("This project involves...", "Based on the analysis...").
- No hedge-padding ("may", "potentially", "it appears that") in titles or summaries. Hedging belongs in `verification_question` only.
- Plain English. If a non-engineer would be confused, rewrite.

## Non-Regression Protocol

### How to make sure we are not making things worse
- Before any change: screenshot or record the current state of every demo surface. Save the current run ID and demo state.
- After any change: compare the same surfaces. If any surface looks worse, the change is rejected until the regression is fixed.
- Never accept "we improved X but Y broke" — both X and Y must be at least as good as before.
- The iteration loop (section above) enforces this: step 6 says "compare the new result to the previous accepted result" and step 7 says "if not improved AND non-regressing, rerun the SAME phase again."

### How to keep iterating until quality is right
- The plan is explicitly designed for looping, not linear progression. Every phase says "if the gate fails, revise and rerun."
- The language quality gate applies to every phase that generates text. It is not a one-time check — it runs every time.
- If a prompt change makes titles better but descriptions worse: revert the description prompt, keep the title prompt, rerun, re-check.
- Track accepted prompt versions. When a prompt passes the quality gate, save it. Only modify from the last accepted version.
- Do not batch prompt changes. Change one prompt at a time, rerun, check quality, accept or reject.

## Architecture Change
- Separate `jira_ticket` records from inferred/proposed `work_item` records. Right now one storage model is trying to hold both, which is why items like `PAW-32` and `Supply Orders` can show up as separate sibling tickets for the same work.
- Persist source-rich ticket payloads in `kb2_tickets`: raw Jira title, raw Jira description, status, assignee, reporter, created/resolved dates, sprint, and normalized summary. The UI should not need to reconstruct rich ticket detail from linked graph nodes.
- Store resolved linked entities directly on ticket docs as stable objects, not just `linked_entity_ids`. Each stored link should carry `node_id`, `display_name`, `type`, and source refs so reruns do not leave the UI with raw UUIDs and empty source panels.
- Keep two explicit state fields on tickets: `semantic_bucket` (`past`, `ongoing`, `proposed`) and `workflow_state` (`backlog`, `todo`, `in_progress`, `review`, `done`). The UI should not have to infer semantic status from a mix of source type and workflow state.
- Add stable dedupe/merge identity for tickets beyond the current sync key so an inferred/proposed ticket can be marked as `same_as` or `derived_from` an existing Jira ticket instead of being materialized as a second visible ticket.
- Make `customer_feedback` -> `proposed_project` -> `proposed_ticket` an explicit model path. The same toy-donation evidence should support both the proposed project and the visible proposed ticket without relying on the UI to guess the relationship.
- Persist explicit evidence packs on proposed projects and proposed tickets. For the toy-donation story, the three feedback submissions should be directly attached as source refs and surfaced in Discover, Tickets, and Guides.
- Make chat strictly tenant-scoped, run-scoped, and demo-state-scoped. `kb2/chat` must read PawFinder data only when the user is on PawFinder; answering from another company is demo-breaking.
- Add a convention-application storage layer that records not just that a convention exists, but how it applies to a feature (`green CTA`, `vertical selection nav`, `client-side small-list browse`). The Guides step should consume these explicit application records instead of inferring from broad page text.
- Ensure human pages and entity pages are generated from the same final scoped state so pages like `Hidden Conventions & Patterns` cannot exist as empty shells while the related entity pages are populated.
- Give verify cards stable semantic identity/dedupe keys so the same issue cannot appear multiple times with slightly different wording or pair ordering. This is especially important for `duplicate_cluster` cards and repetitive low-confidence cards.
- Separate verify-card display fields more cleanly: short display title, detailed problem explanation, explicit verification question, and recommended action. The UI should not force the title to carry the whole speculative sentence.

## Step Add Or Removal
- Add a dedicated deterministic ticket materialization step after Pass 1 Step 9. That step should project graph/entity evidence into ticket docs, enrich them, dedupe them, and stamp semantic bucket/state before any UI reads them.
- Remove ticket creation/sync responsibility from `page-plan.ts` / Step 12-style page planning. Page planning should consume final ticket docs, not create or reinterpret them.
- Add a ticket QA artifact/check after ticket materialization that fails if there are stale linked IDs, missing linked names, missing source refs for Jira-backed tickets, or obvious duplicate sibling tickets for the same work.
- Add a deterministic proposed-ticket materialization step for feedback-backed features. When customer feedback clusters into a feature like Toy Donation, the pipeline should emit at least one human-readable proposed ticket, not only a proposed project.
- Add a deterministic evidence-clustering check that confirms the three toy-donation feedback items are grouped into the same proposed work-item family rather than left as isolated feedback nodes.
- Add a convention-application QA artifact after graph re-enrichment. Fail if `Toy Donation Feature` does not carry usable convention links to Kim, Tim, and Matt when the evidence exists.
- Add a chat/demo readiness check after page and how-to generation. Fail if the Hidden Conventions page is empty, the Toy Donation proposed ticket is missing, or chat answers from the wrong tenant/company.
- Treat Execute as bonus-only in final demo gating. The main acceptance path should end at the generated How-To guide; agent execution should not be required for a run to be considered demo-ready.
- Add a verify-card normalization/dedupe pass after card creation. That pass should collapse semantically duplicate cards, enforce canonical pair ordering for duplicate cards, and remove repeated low-value cards before the UI reads them.
- Add a verify-card title normalization pass or check. Card titles should become short entity-first labels, while hedging and nuance move into `problem_explanation` and `verification_question`.

## Prompt Change
- Change Jira-backed ticket prompts so they preserve concrete source detail. The model may add a short summary, but it must not replace the raw scope with vague text like `A ticket for...`.
- Change inferred/proposed ticket prompts so they first ask whether the work is already represented by an existing Jira ticket, PR, or project. If yes, emit a relation such as `same_as` / `derived_from` instead of a new standalone ticket.
- Change link-generation prompts so linked entities return human display names plus a short reason/evidence string, not only raw IDs.
- Change ticket-writing prompts so priority and state are normalized into explicit enums rather than mixed labels like `P2` in some places and `Medium` in others.
- Change proposed ticket prompts so titles are human-readable and specific (`Toy Donation Feature`, `Dedicated Toy Donation Flow`, etc.), not vague fragments or unrelated backlog labels.
- Change how-to prompts so they explicitly name the hidden conventions they are applying and map each one to a concrete implementation choice: Kim -> green money CTA, Tim -> vertical/sidebar pet-selection flow, Matt -> client-side load for small lists.
- Change human page prompts so `Hidden Conventions & Patterns` is a narrative page with evidence, not an empty placeholder.
- Change chat prompts so answers can be framed for four roles: developer, product manager, project manager, and executive. The answer style may vary by role, but the evidence base must stay the same.
- Change verify-card prompts so they prioritize demo-critical issues: wrong attribution of conventions, missing proposed ticket for feedback-backed work, missing source refs, missing convention applications, or wrong-tenant chat contamination.
- Change verify-card title prompts so titles are concise, entity-first, and non-speculative. Max 10 words. No hedging. Example: "Kim Color Convention — Wrong Attribution".
- Change verify-card prompts so uncertainty and reviewer guidance go into `verification_question` and `problem_explanation`, not into the title itself.
- Change duplicate-card prompts so pair titles are canonical and symmetric (same ordering every time), preventing `A and B` / `B and A` duplicates from both appearing.
- Add explicit prompt instructions for all text-generating steps: "Write like a knowledgeable teammate, not an AI. Use short sentences, active voice, plain English. No filler intros. No hedge words in titles. If a non-engineer would be confused by the text, rewrite it."
