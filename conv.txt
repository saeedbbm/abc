### What I think you should do (high-level)
- **Make “claims + evidence + owner” the atomic unit** (not pages). Pages become fast “views” that can duplicate claims for readability, while **one verification card controls all duplicates**.
- **Make the knowledge graph the routing layer**: it decides **what pages exist**, **what goes on them**, **who owns verification**, and **what to regenerate** on daily updates.
- **Run Architecture 2 as a parallel product slice**: separate storage + separate pipelines + separate UI routes at `/brewandgo2`, fed by the same input endpoint.

---

### Architecture 2 (graph-first) — the backbone
#### 1) Data model (the key objects)
- **InputDoc**: your current flattened input text units (Slack thread, Jira ticket, Confluence page, PR, etc.), with `sourceType`, `sourceId`, timestamps.
- **Entity**: `person/team/system/service/integration/project/customer/process/decision/ticket` (expand later, but start here).
- **GraphEdge**:
  - `MENTIONS` (doc → entity)
  - `RELATES_TO` (entity ↔ entity; typed: `owns`, `depends_on`, `integrates_with`, `part_of`, `decided_in`, etc.)
- **Claim** (most important):
  - `truth_status`: `direct` (explicit in input) vs `inferred` (derived from GitHub patterns / behavior)
  - `confidence`, `severity`, `ownerPersonId`
  - `entity_refs[]`
  - `evidence_refs[]` (doc + span/excerpt pointers)
- **FactGroup / VerificationCard**:
  - clusters multiple near-duplicate claims/occurrences into **one card**
  - actions (validate/edit/reject) apply to all linked occurrences
- **Page + PageItemOccurrence**:
  - pages are **page-owned for performance**
  - each item references a `factGroupId` (so duplicates stay manageable)

This matches what you want: **fast pages for humans + one verification workflow for updates**.

#### 2) Pass 1 (high recall): “Generate everything, but route it correctly”
Graph-first steps:
1. **Snapshot inputs for `/brewandgo2`** (so Arch2 doesn’t depend on Arch1 DB survival later).
2. **Build/refresh graph** from docs:
   - extract entities per doc
   - link doc→entity mentions
   - infer entity↔entity edges (ownership/deps/working-on) using your resolver-style logic + heuristics
3. **Graph-driven page planning (deterministic first, LLM second)**:
   - deterministic rules ensure coverage (e.g., a service entity ⇒ a service page; a decision claim ⇒ decision record index + linked pages)
   - then an LLM can **name/group** pages (titles and clustering), but it must output a strict `PagePlan` schema and you validate constraints (no duplicates, page cap, required categories)
4. **GraphRAG retrieval replaces “flat top-k vector search”**:
   - pick anchor entities for the page (e.g., service X)
   - traverse 1–2 hops to gather: owners, deps, integrations, related decisions, related tickets
   - then do vector search **scoped/seeded by those nodes** (only chunks from docs that mention those entities, plus a small global spillover)
5. **Generate pages with strict templates** (reuse your current templates/Zod style):
   - output structured `PageItems[]` that include `entity_refs`, `truth_status`, `confidence`, and `evidence_refs`
6. **Create verification cards immediately** for:
   - `inferred` claims
   - conflicts
   - low confidence
   - missing/weak evidence
   - and enforce: **every card gets an owner** via graph ownership edges; if still unknown, fallback to a configured “default verifier” person for Eng/PM.

#### 3) Pass 2: “Clean it up, dedupe it, and improve low-confidence”
Pass 2 should operate on **(graph + pass1 pages + claims)**:
- **Dedupe into FactGroups** using:
  - entity_refs overlap (strong signal)
  - semantic similarity embeddings (support signal)
  - evidence overlap (support signal)
- **Conflict detection**: contradicting claims in same group ⇒ one conflict card with clear “what to decide”
- **Citation repair / evidence upgrade**:
  - for low confidence groups, do a second targeted GraphRAG pull (graph neighborhood + scoped vector) to find better direct quotes
- **Temporal stabilization**:
  - tag claims as current/deprecated when timeline evidence exists
- **Cascade updates**:
  - when a group’s canonical claim is updated/validated, regenerate all occurrences on pages (still keeping pages “fast”).

#### 4) Incremental daily updates (lunch + end-of-day)
Graph makes this easy:
- ingest new docs → update graph incrementally
- compute impacted entities → impacted pages (by anchor entity) → regenerate only affected sections
- update/extend FactGroups; create new cards where needed

Critically: **you’re not “regenerating everything”**, you’re regenerating *the set of views touched by new evidence*.

#### 5) “Store every step” (debuggable pipeline)
For every run, store artifacts per step:
- `inputSnapshot`
- `graphBuildResult` (nodes/edges counts + diffs)
- `pagePlan`
- `retrievalPacks` (what context each page saw)
- `pass1Pages`
- `pass2FactGroups + verifyCards`
This is exactly how you find what went wrong.

### Yes — do both, but don’t split into “two separate KBs”
You *can* have **concept-centric pages for humans** (few, merged, editable) and **entity-centric/detail for AI** *without* maintaining two divergent document trees.

The trick is to store one underlying truth layer (claims + graph + evidence), then render **two projections**:

- **Human Projection (Concept-centric)**: fewer pages, merged by topic (e.g., “Observability”, “CI/CD”, “Architecture Overview”), with summaries and curated sections.
- **AI Projection (Entity-centric views)**: one page per `service/system/integration/decision/ticket`, highly structured, exhaustive, optimized for retrieval and agent execution.

Both projections are generated from the same **Claim/FактGroup** backbone, so verification/editing updates both automatically.

---

### Should the AI “use the messy graph” or the pages?
**Use both**, for different roles:

- **Graph is for routing + recall**: pick the right entities, neighbors, owners, related tickets/decisions, and targeted evidence.
- **AI pages are for precision + structure**: when an agent needs an answer, you want it to read a consistent schema (“owner”, “deps”, “runbook”, “SLIs”, “recent decisions”, “risks”), not raw graph edges.

So the AI shouldn’t read the raw graph directly as “content”; it should use the graph to **retrieve** the right structured pages/claims/evidence.

### 1) If users edited all verify cards but still think a page needs changes—how do they tell you?
Add a **“Request change”** action on every page/section/paragraph that creates a *new tracked object* (not a freeform edit that bypasses the truth layer):

- **Request change (gap / rewrite / missing page / wrong placement)** → creates a **Verify card of type `gap`** (or a lightweight “KB task”) with:
  - what looks wrong (user text)
  - what they expected instead
  - where they saw it (page + section anchor)
  - optional: who to ask (mention)
- System then does **targeted GraphRAG** to propose:
  - new claims to add, or claims to modify
  - new evidence candidates
  - impacted pages (where it will propagate)
- User approves → becomes **claims/factgroups updates** → pages regenerate.

This preserves your rule: **edits become claims**, so everything stays debuggable and propagates cleanly.

---

### 2) “One underlying truth layer (claims + graph + evidence), then render two projections” — what that means, concretely
Think of it like a database + two views:

#### The underlying truth layer (what you store)
- **Claims** (atomic statements) + `direct/inferred`, confidence, owner, severity
- **Evidence** (source excerpts and pointers)
- **FactGroups / VerificationCards** (one card controls many duplicates)
- **Graph** (entities + relationships) for routing/retrieval/planning

#### Projection A: Human pages (concept-centric hubs)
- “Observability”, “CI/CD”, “Architecture Overview”, etc.
- These pages are **curated summaries** composed of claim-backed paragraphs.
- They are meant to be readable, not exhaustive.

#### Projection B: AI pages (entity-centric)
- One page per **service/system/integration/decision/ticket/person**
- Highly structured, exhaustive, consistent schema
- What agents retrieve most often

Both projections point back to the **same claims/factgroups/evidence**, so verification updates both.

### 1) Yes — you can have both “edit in place” and “request change” (without breaking the truth layer)
You’re right: forcing a verify-card workflow for *every* tiny fix is annoying.

A good hybrid is:

- **Inline edit (privileged users like team leads)**:
  - User edits the page text directly
  - On **Save**, the system opens a **“Change → Claims”** modal that auto-detects what changed and proposes structured updates:
    - which **claims/factgroups** will be updated/created/retired
    - what **evidence** (if any) supports the new text
    - what **pages/sections** will be impacted
  - User confirms → it becomes claims updates → pages regenerate
  - If evidence is missing, it creates a **verify card** (“needs evidence / human attestation”)

- **Request change (everyone else, and for ambiguous changes)**:
  - Creates a gap/rewrite/wrong-placement card
  - Routes to the owner/team lead for approval

So the page stays editable for the right people, but *no edit bypasses claims*.

---

### 2) Those “anchor entities / traversal path / retrieval pack / factgroups” are about **how the page was generated**, not “one long AI page”
You don’t need one single long exhaustive page.

Instead, the AI uses a **bundle of structured context** assembled at runtime (and stored for transparency). The terms mean:

#### **Anchor entities**
The “main things” the page/section is about.
- Example on a human page section “Checkout reliability”:
  - anchors: `System: Checkout`, `Service: Payments`, `Team: Platform`

These anchors tell the system **where in the graph to start**.

#### **Traversal path**
The exact graph hops used to gather related context.
- Example:
  - `Service: Payments -> owned_by -> Team: Platform`
  - `Service: Payments -> depends_on -> Integration: Stripe`
  - `System: Checkout -> includes -> Service: Cart`

This is how GraphRAG avoids “random top‑k chunks” and pulls the *right neighborhood*.

#### **Retrieval pack**
The concrete inputs that were fed into the LLM to write that paragraph/section/page.
Usually a mix of:
- **Entity pages** (AI-structured, per service/system/decision/ticket)
- **Claims/factgroups** (canonical statements + verification state)
- **Evidence excerpts** (quoted snippets from Slack/Jira/Confluence/GitHub-derived text)
- Sometimes a small number of **vector chunks** found within the graph-scoped neighborhood

Think of it as: “Show me exactly what the model saw.”

#### **FactGroups used + verification status**
This lists the canonical fact clusters that support the text, and whether they’re:
- `validated` (treated as company truth)
- `needs_verification`
- `conflict`
- `inferred`
- etc.

This is what enables “edit once, propagate everywhere.”

---

### 3) So what do AI agents actually read?
They read **entity-centric structured pages** (service/system/decision/ticket pages) *plus* the relevant factgroups/evidence for the task.

Not one mega-page—because mega-pages are slow, noisy, and hard to keep consistent. Entity pages + claim bundles are easier to retrieve, update, and verify.

---

### 4) How this looks in the UI (simple mental model)
On a human concept page (e.g., “Observability Standards”):

- Main content: clean, merged, human-friendly
- A paragraph has:
  - **Sources** tab → “proof”
  - **AI Context** tab → “what the model used to write this”
  - **Open entity view** → jump to `Service: Payments` (AI page) etc.

This gives trust without maintaining two separate documentation trees.

## 2) Core data model (the backbone that enables everything)
Architecture 2 stores one “truth layer” and renders multiple projections.

### Truth layer objects
- **InputDoc**: snapshot of the input items (from the shared input endpoint)
- **Entity**: person/team/system/service/integration/project/customer/process/decision/ticket (+ expand later)
- **GraphEdge**: relationships (ownership, deps, integrates_with, part_of, etc.)
- **Claim**: atomic statement with:
  - `truth_status`: `direct` (explicit in input) vs `inferred`
  - `confidence`, `severity`, `owner`
  - `entity_refs[]`
  - `evidence_refs[]` (source excerpts)
- **FactGroup / VerificationCard**: clusters duplicates/conflicts so **one card controls many occurrences**
- **Page + PageItemOccurrence**: page-owned items for speed, but each occurrence references a `factGroupId` so edits/verification propagate.

This satisfies your key constraint: pages can duplicate for readability, but verification stays centralized.

### Core guardrails (so you don’t get “yapping KB”)
You’re describing two non-negotiables:

1) **Agent usefulness**: templates must cover the *minimum set of facts* that agents + eng/PM need to execute work and make tradeoffs.

- **MUST (always show)**: only for fields that are operationally required for routing work (owner, criticality, repos, linked systems, decision status, etc.). If missing → create verify card.
- **MUST-IF-PRESENT (include only if evidence exists)**: most narrative sections.

## 1) Fixing the “client vs internal people/team” misclassification (be explicit)
You’re right: this is a recurring failure mode. Solve it with **taxonomy + rules**, not hoping the LLM guesses.

### Define “Client/Customer” precisely
- **Customer (B2B)**: an *external company* you sell to or support. May have **contacts** (people) linked to it.
- **User segment (B2C)**: a *group of end-users* (e.g., “iOS users”, “Android users”, “Power users”) — not individual humans.

### Hard rules (template + extractor)
- **A person name is NEVER a “client” entity by itself.** A person can only be:
  - `Person` (internal), or
  - `ExternalContact` (only when linked to a specific B2B customer company)
- **“Client/Customer” pages only list external entities**.
- **If evidence is ambiguous** (“talked to Alex about onboarding”), default to **internal Person** and raise a **verify card** if it might be external.

### Classification cues (fast heuristics you can bake in)
- Internal Person signals: Slack handle, your email domain, “team”, “on-call”, “PR”, “repo”, “standup”, “sprint”, “deploy”.
- B2B Customer signals: company suffixes (Inc/LLC), “contract”, “invoice”, “renewal”, “account”, “POC”, “customer asked”, “support ticket from”.

## 6) Pass 1 / Pass 2 meaning
### Pass 1 (high recall)
- Generate comprehensive structured knowledge quickly.
- It’s okay to be noisy; verification and Pass 2 refine.

### Pass 2 (refine)
- Dedupe into factgroups, detect conflicts, improve low-confidence by re-reading targeted evidence, propagate canonical updates across pages.
- Pass 2 also exists to “take another pass” on low-confidence items to find better evidence.

## 7) Architecture 2 pipeline (step-addressable + reproducible)

### Key principle: snapshot shared input, then run independently
- Step 0 always creates an **Input Snapshot** from `/brewandgo/kb` into Arch2 storage.
- Every other step reads from that snapshot so runs are reproducible.

### Pass 1 steps (example breakdown)
1) **Input Snapshot**
   - store raw input bundle + metadata
2) **Chunking + Embeddings (Arch2 Qdrant collection)**
   - optional early; depends on your existing setup
3) **Entity extraction**
4) **Graph build + relationship resolution**
5) **Gap detection (graph-based)**
6) **Page plan**
   - deterministic rules first (entity-driven)
   - optional LLM for naming/grouping within strict schema
7) **GraphRAG retrieval packs per page**
   - graph neighborhood + scoped vector retrieval
8) **Generate pages**
   - human pages (concept hubs)
   - AI entity pages (service/db/env/decision/ticket/etc.)
9) **Create verify cards**
   - inferred, conflicts, low-confidence, missing owner/evidence

### Pass 2 steps
1) **Cluster to FactGroups**
2) **Conflict detection + verification grouping**
3) **Low-confidence evidence upgrade**
4) **Citation repair / consistency checks**
5) **Propagate canonical updates**
6) **Finalize pass2 outputs**

### Runner controls (must support all options from day 1)
- Run **Pass 1 end-to-end**
- Run **Pass 2 end-to-end**
- Run **one step only** (e.g., Pass 1 step 1 only)
- Run **from step N to end** (“run the rest”)
- **Rerun step N** multiple times while pinning upstream artifacts (step attempts)
- Use cached artifacts automatically when available

This enables fast iteration: fix a prompt/template for step 5, rerun step 5+ without redoing snapshot/entity extraction.
