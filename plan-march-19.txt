# KB2 Pipeline v2 Master Plan — March 19, 2026

## Philosophy

1. **No GT leakage.** Ground truth files are read ONLY by the evaluation system. No pipeline step prompt, quality gate, or pass/fail threshold is derived from GT. Entity count ranges in gates are derived from INPUT statistics (e.g., "6 unique first names in Slack speakers" → expect 6 team members), never from the answer key.

2. **Multi-model adjudication, disagreement-driven.** Use Claude (primary) + OpenAI GPT-4o (cross-check). But don't triple-judge everything — escalate to expensive models only when primary and cross-check disagree. This maximizes quality without burning money on unanimous cases.

3. **Immutability.** Every step writes to its own execution_id. No step deletes or modifies another step's data. Read paths filter by latest execution_id. The "old artifacts" problem is solved by reading correctly, not by deleting.

4. **Evidence ledger.** Every extracted fact, relationship, and attribution stores an immutable evidence span: `{ doc_id, section, char_offset_start, char_offset_end, verbatim_quote_hash }`. This makes every claim auditable without re-reading source documents.

5. **LLM-as-Judge in every step.** Every step gets a judge evaluation that produces a structured report: overall percentage score, sub-scores, reasons, and specific issues. This report is stored in the step's `run_steps` record and displayed in a dedicated "LLM as Judge" tab in the admin UI.

---

## Models

| Role | Model | When Used |
|------|-------|-----------|
| **Primary (fast)** | Claude Sonnet 4 | Most generation: extraction, page generation, edge inference |
| **Reasoning** | Claude Opus 4 | Critical synthesis: conventions, team member pages, how-to guides |
| **Cross-check** | OpenAI GPT-4o | Second opinion on attribution, entity resolution, conventions |
| **Reranker** | Cohere Rerank v3.5 (or LLM fallback) | Step 13 retrieval reranking |
| **Embedding** | Current configured provider | Step 2 embedding |

Escalation rule: Primary generates. If a quality gate flags an issue, cross-check re-evaluates. If primary and cross-check disagree, reasoning model adjudicates. This avoids blanket triple-judging.

# SECTION 1: Steps 1–5 (Ingestion Through Resolution)

---

## Step 1: Input Snapshot

### Goal from goal.txt
Ingest all raw data preserving attribution: who said what, when, in what context. This is the foundation — every downstream attribution error traces back to losing "Kim said X" at ingestion time.

### What Changes

**Conversation-level Slack parsing.** Current: 3 channel documents. New: ~39 conversation documents, one per date-block per channel. Each document stores:
- Channel name, date, participant names (extracted from speaker lines)
- Individual messages with speaker attribution
- This directly enables goal.txt's requirement: "source refs spanning 5 Slack conversations" — you can't trace to individual conversations if they're merged into channel blobs

**Section-aware Confluence parsing.** Preserve heading hierarchy as metadata on each section sub-document. Parent-child links from section back to full document. This enables Step 2 to chunk at section boundaries instead of blindly.

**PR-aware GitHub parsing.** Separate PR description from review comment threads. Each review comment gets the reviewer's name in metadata. This is critical for Matt's browse pattern — the evidence lives in Matt's review COMMENTS, not the PR description. If the review comment doesn't carry Matt's name, Step 9 and Step 10 cannot attribute correctly.

**Jira field preservation.** Ticket status (Done/In Progress/Backlog), assignee, reporter, sprint, resolution date all stored as structured metadata. Step 9 uses these for status inference.

### LLM as Judge — Step 1

**Type: Deterministic only (no LLM needed)**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Slack document count | Count parsed docs where source=slack | 35-45 (derived from scanning date-headers in slack.txt, not from GT) |
| Jira document count | Count parsed docs where source=jira | Must equal number of ticket blocks in jira.txt |
| GitHub document count | Count parsed docs where source=github | Must equal number of PR blocks in github.txt |
| Confluence document count | Count parsed docs where source=confluence | Must equal number of doc blocks in confluence.txt |
| Webform document count | Count parsed docs where source=customerFeedback | Must equal number of submission blocks in webform.txt |
| Speaker attribution | % of Slack docs that have participants[] with ≥1 name | 100% |
| PR reviewer attribution | % of GitHub review comment docs with reviewer name | 100% |

**Overall score:** Weighted average. Pass threshold: 90%.

**How count targets are derived (NO GT leakage):** Parse the raw input files and count structural delimiters (date headers in Slack, ticket blocks in Jira, PR blocks in GitHub, document blocks in Confluence, submission blocks in webform). The expected count comes from the INPUT, not from ground truth.

## Step 2: Embed Documents

### Goal from goal.txt
Enable retrieval that can find "Kim's Slack message from 2023-03-14 about pink accents" when queried about Kim's color choices. Fixed-size chunking destroys this by splitting conversations mid-message.

### What Changes

**Source-type-aware chunking dispatcher:**
- Confluence → split on markdown headings (##, ###). Each section = one chunk. If section > 2000 chars, split at paragraph boundaries.
- Slack → split at message boundaries. 3-5 consecutive messages per chunk. Speaker names preserved in chunk text.
- GitHub → PR description = one chunk. Each review comment thread = one chunk. Reviewer name in metadata.
- Jira → Each ticket = one chunk (tickets are small).
- Webform → Each submission = one chunk.

**Parent-child chunk linking.** Every chunk stores `parent_doc_id` pointing to its full source document. When Step 13 retrieves a short chunk, it can expand to the parent for more context.

**Metadata-prefixed embedding.** Prepend `[slack | #design 2023-03-14 | Kim, Sarah]` to chunk text before embedding. This helps the embedding model distinguish speakers and contexts.

### LLM as Judge — Step 2

**Type: Deterministic + lightweight LLM sample check**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Total chunk count | Count chunks created | > 150 (sanity check — 2788 input lines should produce many chunks) |
| Max chunk size | Longest chunk in characters | < 3000 |
| Parent linkage | % of chunks with valid parent_doc_id | 100% |
| Sentence completeness | LLM reads 20 random chunks, scores each: "does this chunk contain at least one complete sentence?" | ≥ 90% of sampled chunks score yes |
| Speaker preservation | % of Slack chunks that contain at least one speaker name (e.g., "Kim:") | > 95% |

**LLM judge call:** Claude Sonnet (fast model). Sample 20 chunks. Ask: "For each chunk, does it contain a complete thought? Rate 1 (fragment) to 5 (complete, self-contained)." Average ≥ 4.0 = pass.

## Step 3: Entity Extraction

### Goal from goal.txt
Extract every distinct entity from the data. Be aggressive on recall — it's better to over-extract and merge later (Step 5) than to miss entities. But provide real verbatim evidence, not paraphrases.

### What Changes

**Dual-model extraction.** Run two independent passes:
- Pass A: Claude Sonnet (primary) — extracts entities in batches of 3 documents
- Pass B: GPT-4o (cross-check) — extracts from the same documents independently

Merge results: entities found by both models (matched by fuzzy name + source overlap) get high confidence. Entities found by only one model get medium confidence if they have 2+ source_refs, low confidence otherwise.

**Source-type-ordered batching.** Process Confluence first (gives canonical project names), then Jira (structured metadata), then GitHub (code context + review comments), then Slack (conversational), then webform (end-user voice). Pass extracted entities from earlier batches as "already found" context to later batches to reduce duplication.

**Ontology constraints in prompt.** Add rules:
- "A project is a multi-ticket body of work. A single bug fix is a ticket, not a project."
- "A phase of a larger project is NOT a separate project entity unless it has its own distinct Jira tickets assigned to different people."
- "A decision is a deliberate choice between alternatives, not a standard implementation detail."

**Verbatim evidence enforcement.** Prompt requires exact quotes. Post-extraction, validate 20% of evidence_excerpts against source documents (character-level fuzzy match > 80%). Flag fabricated excerpts.

**Evidence ledger entries.** For every extracted entity, store immutable evidence spans: `{ doc_id, section, char_offset_start, char_offset_end, verbatim_quote_hash }`. These are the atomic units of provenance for the entire pipeline.

### LLM as Judge — Step 3

**Type: Deterministic + Dual LLM**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Entity recall (team members) | Count team_member entities vs unique first names found in Slack speakers and Jira assignees | Should match (derive count from input parsing) |
| Entity recall (tickets) | Count ticket entities vs ticket blocks parsed in Step 1 | ≥ 90% coverage |
| Entity recall (PRs) | Count pull_request entities vs PR blocks parsed in Step 1 | ≥ 85% coverage |
| Evidence quality | % of sampled excerpts that fuzzy-match source text (> 80% char overlap) | ≥ 80% |
| Type distribution | At least 1 entity of each core type: project, decision, process, team_member, ticket, pull_request, customer_feedback | All present |
| Cross-model agreement | % of entities found by both Pass A and Pass B | ≥ 50% (high = good recall overlap) |

**LLM judge:** Claude Opus reviews 30 random entities. For each: "Is this a real entity? Is the type correct? Is the evidence genuine? Rate 0-10." Average ≥ 7.0 = pass.

**Cross-check:** GPT-4o reviews the same 30 entities independently. If Opus and GPT-4o disagree on > 5 entities, flag for investigation.

## Step 4: Extraction Validation

### Goal from goal.txt
Catch what Step 3 missed. Challenge weak entities. Pre-seed merge candidates for Step 5 so resolution has a head start.

### What Changes

**Adversarial validation pass.** The reasoning model (Claude Opus) reviews the full entity list and challenges it:
- Fragmentation: "Are 'Pet Profile Cards' and 'Pet Browse Page' the same project? Their Jira tickets overlap."
- Phantoms: "This entity has 1 source ref from a paraphrased excerpt. Is it real?"
- Type errors: "This 'pipeline' entity has Jira tickets and a timeline — should it be a 'project'?"

**Cross-model gap check.** Run gap detection with BOTH Claude Sonnet and GPT-4o on documents that contributed ≤ 1 entity. Merge results:
- Both models agree entity is missing → add with high confidence
- Only one model finds it → add with medium confidence + flag
- Both agree something is phantom → remove
- They disagree on a phantom → keep but flag

**Merge candidate annotations.** Tag entity pairs that should be evaluated for merging in Step 5. Store as `_merge_candidates` on each node: `[{ target_name, reason, confidence }]`. These become pre-seeded candidate pairs in Step 5's blocking phase.

**Cross-reference validation.** For each decision entity, check: does the `decided_by` attribution make sense given who authored/reviewed the PR or who spoke in the Slack message? Flag suspicious attributions early.

### LLM as Judge — Step 4

**Type: Deterministic + LLM (cross-check model)**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Entity count stability | % change from Step 3 count | < 20% change |
| Merge candidates generated | Number of entity pairs tagged for Step 5 | ≥ 10 pairs |
| Source ref coverage | % of entities with ≥ 1 source_ref | 100% |
| Type completeness | At least 1 entity of each core type exists | All present |
| Gap check yield | Number of genuinely new entities added | ≥ 2, ≤ 15 |

**LLM judge:** GPT-4o (use cross-check model since Claude did the primary validation). Compare Step 3 and Step 4 entity lists. "Did validation catch real problems? Did it introduce errors? Score 0-100."

## Step 5: Entity Resolution

### Goal from goal.txt
Merge duplicates into canonical entities. Ground truth expects ~21 projects, ~16 decisions, ~9 processes, ~6 team members. Current run has 43 projects, 31 decisions, 12 processes. The resolution is far too conservative.

### What Changes

**Three-phase resolution:**

**Phase 1 — Blocking (programmatic).** Generate candidate merge pairs using multiple signals. A pair enters the candidate set if ANY signal fires:
- Merge candidates from Step 4's `_merge_candidates`
- Shared Jira tickets in source_refs (if two projects both cite PAW-15, they're likely the same)
- Shared PR numbers in source_refs
- Name token similarity (Jaccard > 0.4)
- Alias overlap (any alias of A matches any alias of B)
- Cross-type pairs where names are very similar (project "API Response Standardization" and decision "Standardized API Envelope Format")

**Phase 2 — Pairwise scoring (LLM).** For each candidate pair, batched in groups of 10-15:
- Claude Sonnet decides: MERGE / KEEP SEPARATE / UNSURE
- For UNSURE pairs, GPT-4o provides a second opinion
- If both disagree, Claude Opus adjudicates

Merge prompt rules:
- "Merge if they refer to the same body of work, even under different names"
- "Merge if one is a component/phase of the other AND the component has no independent Jira tickets assigned to different people"
- "Do NOT merge if one is a decision ABOUT a project — those are separate entities with a relationship"
- "When merging, keep ALL source_refs and aliases from both"

**Phase 3 — Clustering + adjudication.** Apply transitive closure (A merges with B, B with C → {A,B,C} cluster). For each cluster of 3+, a focused Claude Opus call reviews the full cluster and decides the canonical name, type, and merged attributes.

**Post-resolution corrections:**
- Status inference: cross-reference Jira ticket statuses. All linked tickets Done → `completed`. Any In Progress → `active`.
- Type reconciliation: entities with Jira tickets + PRs + timeline = `project`. Choices between alternatives = `decision`. Repeatable workflows = `process`.

### LLM as Judge — Step 5

**Type: Deterministic + Dual LLM judge (this is a critical step)**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Project count alignment | Count project entities | In range derived from input: count distinct multi-ticket initiatives in Jira (programmatic) |
| Decision count alignment | Count decision entities | Reasonable range (no GT-derived number — judge assesses whether decisions are appropriately granular) |
| Process count alignment | Count process entities | Reasonable range |
| Team member count | Count team_member entities | Must match unique first names from Slack speakers + Jira assignees |
| No shared primary tickets | No two project entities share the same Jira ticket as their sole ticket | 0 violations |
| Source ref preservation | Every entity has ≥ 1 source_ref | 100% |
| Status accuracy (sample) | LLM reviews 15 random entity statuses against their source_refs | ≥ 80% correct |

**LLM judge — DUAL:**
- Claude Opus reviews the full entity list: "Are there obvious remaining duplicates? Entities incorrectly merged? Types wrong? Score 0-100."
- GPT-4o reviews the same list independently.
- If they agree (within 15 points): use average as the score.
- If they disagree (> 15 points apart): flag specific disagreements for review. The lower score determines whether to re-run.

**Pass threshold: Average ≥ 60.** (This is advisory, not a hard block — LLM scores are noisy. The deterministic checks are the hard gates.)

## Step 6: Graph Build

### Goal from goal.txt
Create real relationships between entities. "The graph can be traversed from Toy Donation Feature → APPLIES_TO → Kim's Color Convention → PROPOSED_BY → Kim." Current run has 100% dangling edges — this is the #1 bug in the pipeline.

### What Changes

**Remove broken MENTIONED_IN-to-document edges.** The current code creates MENTIONED_IN edges where `target_node_id` is a document sourceId (a string like "PawFinder Website Redesign"), not a graph node UUID. Documents are not graph nodes. This entire code path is wrong.

**Replace with a document mention index (not edges).** Instead of graph edges, store document mentions as a separate lightweight index: `{ node_id, doc_id, mention_count }`. This preserves the provenance information (which nodes are mentioned in which docs) without polluting the graph with invalid edges. The graph should only contain entity-to-entity relationships.

**Generate real entity-to-entity edges.** Two approaches, combined:

1. **From _relationships attributes:** Step 3 extracted `_relationships` on each entity (e.g., `{ target: "Tim", type: "WORKS_ON" }`). Resolve target names to actual node_ids using fuzzy matching (exact → alias → token overlap → substring). Only create the edge if the target resolves.

2. **LLM-inferred edges for co-occurring entities:** For entity pairs that co-occur in the same source document but have no `_relationships` entry, use LLM to infer the relationship type. Batch in groups of 20 pairs. The LLM chooses from: WORKS_ON, REVIEWS, DEPENDS_ON, USES, STORES_IN, DEPLOYED_TO, MEMBER_OF, CONTAINS, PART_OF, APPLIES_TO, PROPOSED_BY, RESOLVES, FEEDBACK_FROM. If no meaningful relationship → NONE (don't create the edge).

**Hard endpoint validation.** Before inserting ANY edge, verify both `source_node_id` and `target_node_id` exist as `node_id` values in `kb2_graph_nodes`. Drop any edge where either endpoint is invalid. This is a hard gate — zero tolerance for dangling edges.

### LLM as Judge — Step 6

**Type: Deterministic (hard gates) + LLM precision check**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| **Dangling edge rate** | % of edges where either endpoint doesn't exist | **0% (HARD GATE — fatal if violated)** |
| Edge type diversity | Number of distinct edge types | ≥ 5 types |
| Entity connectivity | % of entities with ≥ 1 edge | ≥ 50% |
| Average edges per entity | Total edges / total entities | ≥ 1.5 |
| Edge type balance | No single type > 50% of total | Pass/fail |

**LLM judge:** Claude Sonnet reviews 25 random edges. For each: "Is this relationship real? Is the type correct? Is the evidence sufficient? Rate: CORRECT / WRONG_TYPE / FALSE." Precision score = % CORRECT.

**Pass: 0% dangling (hard) AND precision ≥ 75% (advisory).**

## Step 7: Graph Enrichment

### Goal from goal.txt
Add relationships that raw extraction missed. Connect entities through semantic similarity and co-occurrence. Kill generic RELATED_TO edges — everything should have a specific type.

### What Changes

**Semantic similarity edge proposals.** For entity pairs with embedding cosine similarity > 0.7 and no existing edge: propose a candidate edge, use LLM to determine the specific type.

**Co-occurrence enrichment.** Entities in the same Slack conversation, PR review thread, or Jira epic get typed relationship proposals:
- Same PR: author WORKS_ON project, reviewer REVIEWS project
- Same Slack conversation: participants DISCUSSED topic
- Same Jira epic: child tickets PART_OF parent

**Entailment-based precision filter.** Use GPT-4o (cross-check model) to verify proposed edges. For each batch of 20 proposed edges: "Does the source evidence support this relationship? YES / WEAK / NO." Only keep YES and WEAK.

**Upgrade generic RELATED_TO.** After enrichment, for remaining RELATED_TO edges, attempt type upgrade based on entity types:
- team_member → project = likely WORKS_ON
- library → repository = likely USES
- decision → project = likely APPLIES_TO
If upgrade fails, keep the edge but assign lower weight.

### LLM as Judge — Step 7

**Type: Deterministic + LLM precision check (cross-check model)**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Dangling edges | Same hard gate | 0% |
| RELATED_TO ratio | % of edges that are generic RELATED_TO | < 20% |
| Edge growth | % increase over Step 6 edges | ≥ 30% |
| Entity coverage | % of entities with ≥ 1 edge | ≥ 80% |
| Precision (sampled) | LLM reviews 20 random edges for correctness | ≥ 75% |

**LLM judge:** GPT-4o reviews 20 random edges (use cross-check since Claude did the enrichment). "Is this relationship real and correctly typed? Rate: CORRECT / WRONG_TYPE / FALSE."

## Step 8: Project & Ticket Discovery

### Goal from goal.txt
Find undocumented projects: "4 past projects completed but never had a Confluence doc", "5 ongoing projects with no formal documentation", "proposed features from customer feedback (Toy Donation Feature from 3 separate requests)." Discovery nodes should not be bare — they need edges and attributes.

### What Changes

**Evidence-strength tiering.** Discovery requires judgment about confidence:
- **High confidence:** Evidence from 2+ distinct source types (Slack + Jira, GitHub + feedback). OR evidence from 1 strong source type with rich detail (e.g., a PR with description + review thread constitutes strong evidence from 1 type). The "2+ source types" rule should NOT be absolute — a single PR with extensive review comments is strong evidence.
- **Medium confidence:** Evidence from 1 source type with moderate detail.
- **Low confidence / speculative:** Vague mentions, single passing references.

**Customer feedback clustering.** Before the LLM call, programmatically cluster feedback by topic similarity (embedding cosine > 0.6). Each cluster becomes one proposed feature with all submissions as source_refs. The 3 toy donation requests → 1 "Toy Donation Feature" entity.

**Status inference from evidence.** Prompt rules:
- Discovered entity with merged PR + Done ticket → `completed`
- Discovered entity with In Progress ticket → `active`
- Discovered entity from customer feedback only → `proposed`
- Do NOT default everything to `proposed`

**Duplicate check against existing entities.** Before creating a discovery node, compute similarity against Step 5 entities. If overlap > 60% (name + source_refs), enrich the existing entity instead of creating a duplicate.

### LLM as Judge — Step 8

**Type: Deterministic + LLM value assessment**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Discovery count | Total new entities | Range [5, 20] (too few = missed things, too many = noise) |
| Evidence strength | % of discoveries with ≥ 1 source_ref | 100% |
| Feedback clustering | No duplicate feature proposals from same feedback cluster | 0 duplicates |
| Overlap check | No discovery duplicates an existing Step 5 entity (name overlap > 70%) | 0 violations |
| Toy Donation found | A proposed feature entity related to toy donation exists | Yes |

**LLM judge:** Claude Opus reviews all discoveries. For each: "Is this a VALUABLE discovery (real entity the extraction missed) or NOISE (redundant, speculative, too granular)? Rate VALUABLE / NOISE." Score = % VALUABLE.

**Pass: ≥ 60% VALUABLE.**

## Step 9: Attribute Completion

### Goal from goal.txt
"Every entity has status, documentation_level, and description. Every decision has decided_by, rationale, scope. Step 8 discovery nodes have the same attribute coverage as Step 5 entities." Currently: 40% status fill, wrong decided_by attributions, only 1 LLM call for 214 entities.

### What Changes

**Fix the 0-write bug.** The clone-and-modify logic must correctly clone nodes from Step 5 + Step 8 into the new execution_id, then modify the clones. Debug and fix the specific point where the insert silently produces 0 documents.

**Per-type batching.** Instead of 1 LLM call for all entities:
- Projects: batches of 10-12, with linked Jira ticket statuses and PR merge dates as context
- Decisions: batches of 6-8, with the source documents that discuss the decision
- Processes: batches of 10
- Team members: 1 batch (only 6)
- Other types: batches of 15

This gives the LLM focused context per batch instead of a saturated context window.

**Corrective inference, not just fill-missing.** The LLM may CORRECT existing status values when evidence contradicts them. Prompt:
- "If ALL linked Jira tickets are Done and ALL linked PRs are merged → status: completed, EVEN IF the entity currently says active"
- "If the entity says proposed but there is evidence of actual work (done tickets, merged PRs) → CORRECT to completed or active"

**Attribution from PR metadata.** For decided_by on decisions, the prompt explicitly instructs:
- "A PR REVIEWER who gives architectural guidance in review comments is the decision maker for decisions stated in those comments."
- "A DESIGNER who says 'I'm making this green' in Slack is the decision maker for that design choice, even if a developer implements it."
- "The person who IMPLEMENTS someone else's design is NOT the decision maker."

Use Claude Opus for decided_by inference (reasoning model — this is critical). For the 5 most important decisions (those likely to become conventions), also run GPT-4o and compare.

### LLM as Judge — Step 9

**Type: Deterministic + Dual LLM (critical step for attribution)**

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| **Node count in new execution** | Count nodes with new execution_id | **> 0 (HARD GATE — fatal if 0)** |
| Status fill rate | % of entities with non-null status | ≥ 80% |
| Description fill rate | % of entities with description (not just _description) | ≥ 90% |
| Documentation level fill rate | % with documentation_level | ≥ 85% |
| decided_by fill rate (decisions) | % of decision entities with decided_by | ≥ 80% |
| Status-ticket consistency | % of entities where status aligns with linked Jira ticket status | ≥ 85% |

**LLM judge — DUAL (focused on attribution):**
- Claude Opus reviews 10 decision entities: "Is decided_by correct given the source evidence? Is status correct? Score each entity 0-10."
- GPT-4o reviews the same 10 independently.
- **KEY CHECK:** If the two models disagree on decided_by for ANY decision, that decision is flagged for a third adjudication call. This is the disagreement-driven escalation pattern — it catches the Matt/Tim attribution problem.

**Pass: Average entity score ≥ 7/10 from both models. Any disagreement on decided_by triggers re-evaluation.**

## Step 10: Pattern Synthesis

### Goal from goal.txt
THE DEMO MONEY SHOT. Discover 3 hidden conventions:
1. Kim's Color Convention — pink/blue for gender, green for money CTAs
2. Tim's Layout Convention — vertical sidebar for selection, horizontal for comparison
3. Matt's Client-Side Browse Pattern — load all for <20 items, paginate for 30+

Each must have: correct attribution, complete pattern_rule (all aspects), 5-10+ source_refs.

### What Changes

**Phase 1 — Programmatic candidate clustering.** Group decisions by decided_by. Within each person's decisions, find clusters of semantically similar decisions. If the same person made 3+ similar decisions over time → candidate convention.

**Phase 2 — Per-cluster evidence collection.** For each candidate cluster:
- Collect ALL source documents from ALL constituent decisions' source_refs
- Also search Qdrant for additional relevant documents (embed the cluster description, retrieve top-10)
- Include FULL relevant sections from source docs, not truncated to 1500 chars

**Phase 3 — Per-cluster synthesis.** One focused Claude Opus call per cluster. The prompt provides:
- All constituent decisions with full source excerpts
- All relevant source documents (full sections)
- Explicit instruction: "If this convention has multiple aspects (e.g., color for gender AND color for money), describe ALL aspects in the pattern_rule. Do not describe only part of the pattern."
- Explicit attribution instruction: "The originator is the person who AUTHORED the PR introducing the pattern, or who WROTE the review comments prescribing it, or who is described as the designer. NOT the person who implements."

**Phase 4 — Attribution verification.** Separate GPT-4o call per convention:
- "Based on the evidence, is {person} the correct originator? Look at PR authorship, review comment authorship, and who proposed the approach in Slack/Confluence."
- If GPT-4o disagrees with Claude Opus: Claude Opus adjudicates with both perspectives presented.

**Phase 5 — Source ref aggregation.** Convention inherits ALL source_refs from ALL constituent decisions, plus any additional sources found during evidence collection. Deduplicate by doc_id + excerpt.

**Documented vs undocumented classification.** If the convention is fully described as a rule in a Confluence doc → `documentation_level: "documented"` (not a "hidden" pattern). If only described per-page but never as a general rule → `"partially_documented"`. If no documentation at all → `"undocumented"`.

### LLM as Judge — Step 10

**Type: Deterministic + Triple LLM (most critical step in the pipeline)**

This step gets the most rigorous judge because it directly determines the demo outcome.

| Sub-score | How Measured | Target |
|-----------|-------------|--------|
| Convention count | Number of conventions with is_convention=true | ≥ 2 (expecting 3 hidden + potentially 1-2 documented) |
| Constituent count | Each convention has ≥ 3 constituent decisions | 100% |
| Source ref count | Each convention has ≥ 5 source_refs | 100% |
| Attribution format | Each convention has established_by filled | 100% |

**LLM judge — TRIPLE (disagreement-driven):**

First: Claude Opus evaluates each convention:
- "Is the pattern_rule complete? Does it capture ALL aspects?" (0-10)
- "Is established_by the correct person based on the evidence?" (yes/no + reasoning)
- "Are the source_refs sufficient to prove this convention?" (0-10)

Second: GPT-4o evaluates the same conventions independently with the same questions.

Third (only if needed): If Opus and GPT-4o disagree on attribution for ANY convention, Claude Opus gets a focused adjudication call where both perspectives are presented: "Model A says the originator is Tim because... Model B says the originator is Matt because... Based on the raw evidence, who is correct?"

**Score calculation:**
- Pattern completeness: average of both models' 0-10 scores / 10 → percentage
- Attribution accuracy: both agree correct = 100%, one says wrong = 50% (flag for adjudication), both say wrong = 0% (critical failure)
- Evidence sufficiency: average of both models' 0-10 scores / 10 → percentage
- Overall = weighted: attribution 40%, completeness 35%, evidence 25%

**Pass: Overall ≥ 70%. Attribution accuracy = 100% (after adjudication if needed). Any attribution failure triggers re-synthesis with corrected guidance.**

## NEW Step 10.5: Convention Validation Gate

### Goal
Catch and correct any remaining attribution errors, pattern incompleteness, or source coverage gaps BEFORE conventions are wired into the graph (Step 11) where errors become permanent.

### What It Does

## Step 13: GraphRAG Retrieval

### Goal from goal.txt
Retrieve rich context for each entity so page generation has the evidence it needs. "Kim's Color Convention has source refs spanning 5 Slack conversations, 3 Confluence doc sections, and 2 PRs." The retrieval must find all of these.

### What Changes

**Alias-aware document matching.** Search for entity display_name AND all aliases when scanning documents for mentions.

**Cross-encoder reranking.** Retrieve top-30 vector results, rerank with Cohere Rerank (or LLM-based reranking as fallback), keep top-5. This dramatically reduces noise.

**Multi-hop graph traversal for conventions.** For convention entities:
- Traverse convention → CONTAINS → constituent decisions → collect their source_refs
- Traverse convention → PROPOSED_BY → team member → collect their related work
- Traverse convention → APPLIES_TO → features → collect feature context
This ensures convention pages get deep evidence chains.

**Parent expansion.** When a retrieved chunk is < 200 chars, also fetch its parent section/document for more context.

## Step 17: Extract Claims

### Goal from goal.txt
Extract verifiable claims from entity pages. Each claim should be traceable to source documents. Claims should be classified as direct (stated in sources) or inferred (synthesized by the pipeline).

### What Changes

**Classification based on provenance, not entity type.** The other LLM correctly pointed out that "convention = always inferred" is too coarse. Instead:
- Claim has 2+ source_refs with valid evidence spans → `direct`
- Claim has 1 source_ref → `direct` (medium confidence)
- Claim has 0 source_refs or confidence = "low" → `inferred`
- Claim about a convention's pattern_rule with supporting evidence → `direct` (the evidence directly shows the behavior)
- Claim about a convention's existence as a pattern → `inferred` (the synthesis is the pipeline's contribution)

**Contradiction detection.** Within each entity's claims, use embedding similarity to find claims about the same topic. For similar pairs, LLM check for contradiction. Flag contradicting pairs for verify cards.

**Deduplication.** Claims from entity pages and human pages that say the same thing should be merged, not duplicated.

### LLM as Judge — Step 17

| Entity linkage | Every claim has at least 1 entity_id | 100% |

**LLM judge:** Claude Sonnet (fast). Sample 30 claims. For each: "Is the type (direct/inferred) correct given the evidence? Rate CORRECT / INCORRECT." Score = % CORRECT.

## NEW Step 19: Pipeline Evaluation Report

### Goal
The comprehensive evaluation that uses ground truth. This step is a SEPARATE evaluation job, NOT part of the production pipeline execution path. It runs after the pipeline completes, reads GT files + all pipeline output, and produces a structured quality report.

### What It Evaluates

**Layer 1 — Documented knowledge, organized (weight: 25%)**
- Entity coverage: for each GT entity, is there a matching pipeline entity? (fuzzy name match)
- Attribute completeness: status, description, documentation_level fill rates
- Page quality: entity pages have real content, source refs, sections

**Layer 2 — Undocumented knowledge, discovered (weight: 30%)**
- Are the 4 past undocumented projects found?
- Are the 5 ongoing undocumented projects found?
- Is the Toy Donation Feature proposed from customer feedback?
- Do discovery entities have edges (not floating islands)?

**Layer 3 — Hidden patterns, synthesized (weight: 45%)**
- Are 3 conventions discovered?
- Attribution accuracy: Kim=color, Tim=layout, Matt=browse?
- Kim's convention has BOTH aspects (gender colors + money colors)?
- Source ref count per convention (target: 5-10+)
- Toy Donation how-to references all 3 conventions with SPECIFIC guidance?
- Traceability: can you traverse Toy Donation → convention → team member?
- Hidden conventions human page exists with real content?

### Evaluation Methods

**Automated metrics (no LLM):**
- Entity count per type vs GT counts
- Entity coverage: for each GT entity, fuzzy-match against pipeline output
- Graph health: dangling %, connectivity, edge type diversity
- Attribute fill rates
- Page metrics: entity_ref resolution rate, section counts

**LLM judges (dual model — Opus + GPT-4o):**
- Sample 10 entity pages, compare against GT descriptions
- Evaluate Toy Donation how-to against GT ideal output
- Evaluate each convention against GT convention specs
- For each evaluation, both models score independently
- Disagreements are reported alongside the scores
