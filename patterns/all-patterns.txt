CATEGORY 1: IMAGE LOADING
================================================================================


--- PATTERN IMG-1: Lazy loading for pet photos ---

Detail:       Pet card images use lazy loading so below-the-fold images are
              not fetched until scrolled into view.
Sources:
  - GitHub PR #18 description: "Lazy loading for pet photos"
  - Jira PAW-4 description: "Lazy loading for images"
Established by: Tim (PR #18 author)


--- PATTERN IMG-2: Skeleton loading states ---

Detail:       Cards and list items show a skeleton placeholder (gray shapes
              mimicking the final layout) while data is loading. Applied across
              multiple features — not a one-off.
Sources:
  - GitHub PR #18 description: "Skeleton loading state"
  - GitHub PR #55 review: Jill adds "skeleton loader for the shift list" on
    Tim's guidance
Established by: Tim (pattern), Jill (extended to volunteer portal)


--- PATTERN IMG-3: Species-specific image fallback ---

Detail:       When a pet photo returns a 404, an onError handler shows a
              placeholder silhouette based on the animal type (dog, cat,
              rabbit, etc.) instead of a broken image icon.
Sources:
  - Jira PAW-26: "Fix pet card image fallback"
  - GitHub PR #65: "onError handler to the pet card image that shows a
    placeholder silhouette based on the animal type"
Established by: Tim (PR #65 reviewed by Tim)

CATEGORY 3: COMPONENT ARCHITECTURE AND PERFORMANCE
================================================================================


--- PATTERN COMP-1: React.memo + useCallback for card grids ---

Detail:       When rendering a grid of cards with click handlers (e.g. Sponsor
              button), wrap the card component in React.memo and use useCallback
              for the click handler to prevent the entire list from re-rendering
              when a single card's button is clicked.
Sources:
  - GitHub PR #18 review: Matt says "the grid re-renders the entire list when
    a single card's sponsor button is clicked. Can you memoize the card
    component?" Tim responds "Fixed with React.memo and a useCallback for
    the click handler."
Established by: Matt (identified the problem), Tim (implemented the fix)


--- PATTERN COMP-2: Promise.all for parallel API calls ---

Detail:       When fetching data from multiple independent endpoints, use
              Promise.all to parallelize the requests instead of awaiting
              them sequentially.
Sources:
  - GitHub PR #43 review: Matt says "you're making a separate API call for
    each shelter. consider using Promise.all to parallelize the requests
    instead of awaiting them sequentially." Jill fixes it.
Established by: Matt (code review pattern)


--- PATTERN COMP-3: Shared Toast component ---

Detail:       A reusable Toast component exists in the shared components folder
              for displaying success/error feedback on forms and actions. Any
              new feature with user-facing feedback should use it.
Sources:
  - GitHub PR #39 review: Tim says "The toast component is great, let's move
    it to the shared components folder so we can reuse it elsewhere."
Established by: Jill (built it), Tim (promoted to shared)


--- PATTERN COMP-4: 200ms CSS fade transition for content switching ---

Detail:       When switching between items in a browse/selection view (e.g.
              flipping between pet cards), use a 200ms CSS fade transition
              on the card container.
Sources:
  - GitHub PR #23 review: Tim suggests "subtle fade or slide" for pet
    transitions. Matt responds "Added a quick CSS transition on the card
    container, 200ms fade."
Established by: Tim (suggested), Matt (implemented)

================================================================================


--- PATTERN NAV-1: Vertical left sidebar for selection ---

Detail:       When the user is choosing or selecting from a list (picking one
              item to drill into), use a vertical menu or sidebar on the left
              side. Not tabs across the top.

    category list as a vertical menu on the left side...the user is choosing,
    not comparing, so vertical nav makes more sense"
  - Slack #engineering 2025-02-17: Tim on resource library: "put the
    categories on the left as a vertical list, not tabs on top"
  - GitHub PR #31: Tim's shelter locations page: "The shelter list is a
    vertical scrollable panel on the left side"
  - GitHub PR #55 review: Tim tells Jill "Put the shelter list on the left
    as a vertical nav" (changed from tabs)
Established by: Tim
Applied on: Pet categories page, shelter locations page, volunteer portal
            shelter selector, resource library page

              columnar layout across the page — items as columns, attributes
              as rows.

    shelters laid out left to right...when you're comparing things, your eyes
    want to go left-right"

Established by: Tim
Applied on: Shelter comparison page

  - GitHub PR #55: "Registration form — name, email, phone...Single column

    single column, clear field labels"
Established by: Tim

CATEGORY 5: API DESIGN
================================================================================


--- PATTERN API-1: Response envelope { data, meta } ---

Detail:       All API endpoints return responses in the shape
              { data: T, meta: { timestamp, requestId } }. A response wrapper
              middleware handles this automatically.
Sources:
  - GitHub PR #27 (Matt's PR that standardized it)
  - Confluence onboarding guide: documents the envelope format
  - Slack #engineering 2025-11-10: Jack confirms "same envelope as everything
    else right? { data, meta }" Matt: "yep. use the response wrapper
    middleware"
Established by: Matt
Applied on: All pawfinder-api endpoints


--- PATTERN API-2: No pagination for small lists (under ~20 items) ---

Detail:       For browse/list features where the dataset is small (under ~20
              items), load the entire list in a single API call on page mount.
              Handle navigation entirely client-side. No server round-trip
              per click.
Sources:
  - GitHub PR #23: Matt's adoption chooser — "8-15 adoptable pets...no reason
    to round-trip to the server for each click"
  - GitHub PR #47 review: Matt tells Tim "Just load them once...only like
    10-12 adoptable pets"
  - GitHub PR #49 review: Matt tells Jack "just load them all upfront...5-8,
    maybe 12 max"
  - GitHub PR #52 review: Matt says "15 shifts is nothing"
  - GitHub PR #55 review: Matt approves "no need to paginate 10-15 items"
Established by: Matt
Applied on: Adoption chooser (8-15 pets), donation page pet dedication
            (10-12 pets), shelter events browser (5-12 events), volunteer
            shift list (10-15 shifts)


--- PATTERN API-3: Cursor-based pagination for larger lists ---

Detail:       When pagination is needed (lists of ~30-40+ items with growth
              potential), use cursor-based pagination (not offset-based).
              Default page size is 20. Cursor is based on the item's
              created_at timestamp + ID for stability.
Sources:
  - GitHub PR #58: "Used cursor-based pagination...Page size defaults to
    20...Cursor is based on the item's created_at timestamp + ID so it's
    stable"
  - GitHub PR #58 review: Matt says "for 30-40 items that's borderline but
    probably fine since it could grow"
Established by: Jack (implemented), Matt (approved)


--- PATTERN API-4: Parse DB strings into structured objects ---

Detail:       Raw strings from the database (e.g. hours of operation stored
              as "Mon-Fri 9am-5pm") must be parsed into structured objects
              (e.g. { day, open, close }) in the service layer before sending
              in the API response. Frontend should never do string parsing.
Sources:
  - GitHub PR #35 review: Matt tells Jack "Parse them into a structured
    object { day, open, close } so the frontend doesn't have to do string
    manipulation"
  - Slack #engineering 2024-03-18: Matt: "don't just pass through the raw
    string from the DB"
Established by: Matt


--- PATTERN API-5: ISO 8601 for all API dates ---

Detail:       All API responses must return dates in ISO 8601 format.
              Normalization happens in the service layer via a utility function.
Sources:
  - Slack #engineering 2024-08-19: Matt tells Jack "yes, always return ISO
    8601 from the API. normalize in the service layer"
Established by: Matt


--- PATTERN API-6: Idempotent DELETE endpoints ---

Detail:       DELETE endpoints should return 200 even if the resource is
              already deleted or absent. No error on double-delete.
Sources:
  - GitHub PR #52 review: Matt tells Jack "the DELETE for shift cancellation
    should be idempotent. If someone double-clicks cancel, it shouldn't
    error. Just return 200 if they're already not signed up."
Established by: Matt


--- PATTERN API-7: Stay on /api/v1 ---

Detail:       New endpoints go under /api/v1. No v2 until there are breaking
              changes. Just add to the existing namespace.
Sources:
  - Slack #engineering 2025-11-10: Matt tells Jack "v1 is fine. we don't
    need v2 until we have breaking changes. just add to the existing
    namespace"
Established by: Matt


--- PATTERN API-8: Validate uniqueness before insert ---

Detail:       Check uniqueness constraints (e.g. duplicate email) in the
              handler before inserting into the database. Return a proper 409
              Conflict response. Do not rely on DB constraint errors for user
              feedback.
Sources:
  - GitHub PR #52 review: Matt tells Jack "The volunteer registration should
    validate email uniqueness before creating the record. Right now it'll
    throw a DB constraint error which isn't a great user experience."
Established by: Matt


--- PATTERN API-9: Capacity check with 422 response ---

Detail:       For limited-availability resources (e.g. shift signups), check
              capacity at signup time in the handler and return 422 if full.
              Optimistic locking approach.
Sources:
  - GitHub PR #52 review: Matt says "check if the shift is full before
    allowing signup"
  - Slack #engineering 2024-11-04: Matt says "check capacity again in the
    signup handler and return a 422 if it's full. optimistic locking
    basically"
Established by: Matt

              This is a small, bounded list.

Detail:       The following pet API endpoints already exist:
              - GET /api/v1/pets (list with filters for species, gender, age)
              - GET /api/v1/pets/:id (individual pet profile)
              - GET /api/v1/pets/adoptable (adoptable pets only)
