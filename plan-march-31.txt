## Full Cheating Audit

Every instance where pipeline code leaks ground-truth answers, hardcodes PawFinder-specific names, or gives hints the system should discover on its own.

### CHEAT 1: generate-howto.ts — hardcoded convention + person names in prompt
- **Line 45-46**: `"Name the person who established each convention. Do not say 'established team patterns' — say 'Kim's color convention', 'Tim's layout convention', 'Matt's browse pattern'."`
- **Severity**: Critical. Directly tells the LLM what to output.
- **Fix**: Replace with: `"Name each convention and its owner using the entity names and person names from the Convention Constraints provided. Do not use vague phrases like 'established team patterns' — use the specific names from the data."`

### CHEAT 2: generate-human-pages.ts — hardcoded example with Kim
- **Line 256**: `"Say 'Kim established the color convention' not 'a team member established a pattern'."`
- **Severity**: High. Tells the LLM both the person and the convention name.
- **Fix**: Replace with: `"Name people and conventions specifically using the data from the AI entity pages. Do not use vague phrases like 'a team member established a pattern' — use the actual names from the data."`

### CHEAT 3: generate-human-pages.ts — PawFinder used in formatting example
- **Line 249**: `"PawFinder connects adopters with local shelters."`
- **Severity**: Low. This is a formatting example showing bullet point structure. The company name appears because it's demonstrating the desired format. The LLM would see PawFinder in the data anyway.
- **Fix**: Replace with a generic company example: `"Acme Corp provides B2B logistics services."` to avoid any appearance of leaking.

### CHEAT 4: create-verify-cards.ts — "Kim Color Convention" in title example
- **Line 414**: `"Example: 'Kim Color Convention — Wrong Attribution'"`
- **Severity**: Medium. Hints at which conventions and people exist.
- **Fix**: Replace with a generic example: `"Example: 'PostgreSQL — Missing Connection Config'"` (this example already exists on line 416, so just remove the Kim one).

### CHEAT 5: create-verify-cards.ts — "Kim's full name" and "Confirm with Kim" in content examples
- **Line 424**: `"Example: 'Kim's full name' not 'The full name of the team member Kim'"`
- **Line 426**: `"Example: 'Confirm with Kim' not 'It is recommended that a team member reach out to Kim to confirm...'"`
- **Severity**: Low-medium. Uses Kim as a shorthand example but doesn't leak convention logic.
- **Fix**: Replace with generic names: `"Example: 'Owner's full name'"` and `"Example: 'Confirm with the team lead'"`

### CHEAT 6: entity-extraction.ts — hardcoded "Client-side browse loading rule" label
- **Line 425**: `label: "Client-side browse loading rule"`
- **Severity**: Medium. This is a deterministic pattern signal that gets injected when `CLIENT_PATTERN_RE` matches. The label names the convention.
- **Fix**: Replace with a generic label derived from the match: `label: "Data loading pattern signal"`. The downstream steps should name it from evidence.

### CHEAT 7: entity-extraction.ts — hardcoded regex patterns that encode ground-truth
- **Lines 56-60**: `COLOR_PATTERN_RE`, `LAYOUT_PATTERN_RE`, `CLIENT_PATTERN_RE` — regex patterns that match the exact keywords from the three ground-truth conventions.
- **Severity**: Borderline. These are feature-detection heuristics, not prompt leaks. They help the extraction step find patterns in raw text. Without them the LLM might still find these patterns, or might not. The issue is that they are specifically tuned to the three PawFinder conventions, making the system less general.
- **Fix**: Keep for now — these are defensible as generic pattern-detection heuristics (color words, layout words, data loading words). But document them as "pattern detection hints" and note they should be generalized for non-PawFinder companies. Flag for future work.

### CHEAT 8: graph-enrichment-v2.ts — hardcoded convention family IDs and titles
- **Lines 66-118**: `detectPatternFamilies()` function hardcodes three convention families:
  - `semantic_color_ui` / "Semantic Color Coding for Pet UI"
  - `selection_layout_ui` / "Vertical Navigation for Pick-One Browse Flows"
  - `client_side_browse` / "Client-side filtering for small lists, pagination for scalable lists"
- **Severity**: Critical. These are the three ground-truth conventions, hardcoded with specific titles and pattern rules. The system doesn't "discover" them — it pattern-matches text against pre-defined regex and emits pre-written convention names.
- **Fix**: This is the hardest one. The `detectPatternFamilies` function is doing what the LLM should do in pattern-synthesis. Two options:
  - **Option A (clean)**: Remove the hardcoded families. Let the LLM in pattern-synthesis (step 10) discover and name conventions from clustered signals. This is the honest path but risks the LLM missing patterns.
  - **Option B (pragmatic)**: Keep the regex detection but make the titles and rules generic: emit a signal like `{family_id: "color_ui_pattern", title: "Color-based UI pattern"}` and let the LLM in pattern-synthesis name it properly from evidence. This preserves the detection power without pre-naming the answer.
  - **Chosen**: Option B for now. Keep the detection regexes but remove specific titles/rules. Let downstream steps name them.

### CHEAT 9: graph-re-enrichment-v2.ts — hardcoded convention name matchers and synthetic decisions
- **Lines 56-75**: `isColorConventionName()`, `isClientSideConventionName()`, `isVerticalNavigationConventionName()` — functions that match convention nodes by hardcoded name fragments.
- **Lines 148-189**: `buildSyntheticDecisionSpecs()` — creates synthetic decision nodes with hardcoded names like "Pet Card Gender Accent Decision", "Financial CTA Color Decision", "Vertical Sidebar Layout Decision", "Load Small Lists Client-Side Decision" and hardcoded summaries.
- **Lines 192-206**: `scoreConventionFit()` — uses convention-specific keyword lists to match conventions to features.
- **Severity**: Critical. The re-enrichment step generates pre-named synthetic entities and uses hardcoded logic to wire them to features.
- **Fix**: Same approach as Cheat 8 — keep the structural wiring logic but remove hardcoded names. The `buildSyntheticDecisionSpecs` function should derive decision names from the convention's `pattern_rule` and `source_refs` instead of hardcoded strings. The `scoreConventionFit` function should use the convention's own description text for keyword matching instead of per-convention keyword lists.

### CHEAT 10: discovery-v2.ts — "Toy Donation Feature" used as example in prompt
- **Line 51**: `"Good: 'Toy Donation Feature'. Bad: 'Feature for Allowing Users to Donate Toys to Shelter Pets'."`
- **Severity**: Medium. Directly names the feature the system should discover.
- **Fix**: Replace with a generic example: `"Good: 'Dark Mode Toggle'. Bad: 'Feature for Allowing Users to Switch Between Light and Dark Display Modes'."`

### CHEAT 11: attribute-completion.ts — PawFinder-specific examples in description prompt
- **Line 555-557**: Good examples include "volunteer registration, shift browsing" and "Kim chose green for all money-related buttons (Donate, Sponsor Me)".
- **Severity**: Medium. The Kim example directly references the color convention. The volunteer example references a specific PawFinder project.
- **Fix**: Replace with generic examples that demonstrate the desired format without referencing PawFinder specifics:
  - `"Backend REST API endpoints for user registration, session management, and profile updates. Requirements cover OAuth integration and rate limiting."`
  - `"Responsive grid layout displaying item cards with thumbnails, titles, and prices. Users can filter by category, sort by date, and paginate results."`
  - `"The design lead chose a consistent accent color for all payment-related CTAs to create a clear visual signal across the platform."`

## Other Issues To Fix

### Issue 2: Chat gives bad answers — tells users to "go talk to Kim"
Root cause: chat RAG retrieval in `app/api/[companySlug]/kb2/chat/route.ts` is broken in two ways:
1. Entity/human page loading (lines 182-183) grabs first N pages without execution scoping or relevance filtering. Convention entity pages often aren't in those first 20.
2. Graph search (line 106-115) is keyword-only. "Conventions" doesn't match node names like "Semantic Color Coding for Pet UI".

**Fix**:
- Add graph edge traversal: when a node like "Toy Donation Feature" is matched, follow APPLIES_TO, PROPOSED_BY, CONTAINS edges to pull convention nodes into context.
- Scope entity/human page queries to latest execution (like howto/verify already do).
- Add the system prompt rule: "Never tell the user to go ask someone else. Answer from the knowledge base or say you don't have the information."
**Rerun**: None needed — this is an API-only change.

### Issue 3: Sidebar groups in How-To page are not collapsible
The left sidebar in `/pawfinder2/howto` shows groups like "Past Tickets", "Ongoing Tickets", etc. but they're always expanded.

**Fix**: Replace `SidebarSection` component with a `Collapsible`-based version. Each group header toggles show/hide. Apply to all three tabs (Tickets, Projects, How-to).
**Rerun**: None — UI-only.

### Issue 4: Right panel shows all sources, not text-specific sources
When viewing a how-to guide, ticket, or project, the right panel dumps all sources at once. Clicking a specific paragraph or section should highlight/filter to just the sources relevant to that text.

**Fix**: Add section-level source mapping:
- For how-to: each section already has `linked_entity_ids`. When a user clicks a section, filter `rightPanelSources` to sources from that section's linked entities only.
- For tickets: clicking the description filters to description-relevant sources. Clicking a source quote highlights it.
- For projects: clicking an entity page section filters to that section's source_refs.
- Add a `selectedSection` state that tracks which section/paragraph is active, and filter `rightPanelSources` accordingly. When nothing is clicked, show all sources (current behavior as fallback).
**Rerun**: None — UI-only.

### Issue 5: Ticket descriptions too short
Descriptions come from step 9 (attribute-completion.ts) which says "1-3 sentences". This is not enough for a demo.

**Fix**: Change step 9 prompt to require 3-5 sentences. The description should cover: what the work is, why it matters, what the key technical decisions are, and what depends on it. Keep the banned patterns.
**Rerun**: Covered by full pass 1 rerun.

### Issue 6: How-to clicking should show source and be editable
The how-to sections are already editable on click (lines 738-749 in KB2HowtoPage.tsx toggle to a textarea). But the source connection is missing — clicking a section does not update the right panel.

**Fix**: Covered by Issue 4 above (section-level source mapping). The edit-on-click already works.

### Issue 7: Customer feedback cards on Discover page show truncated messages
The Discover page only shows a partial excerpt of each customer feedback submission. The full message should be visible both in the card itself and in the right panel source view.

**Fix**:
- In the Discover page component (`KB2DiscoverPage.tsx`), render the full feedback body text on each card instead of truncating.
- In the right panel, when a feedback card is selected, pass the full text as the source excerpt rather than a sliced substring.
- Check where the truncation happens: likely the API route slices `excerpt` or the component truncates with CSS/JS. Remove the truncation for feedback items.
**Rerun**: None — UI and possibly API route fix only.

### Issue 8: Sources and Linked Entities still shown inline on Guides ticket/project detail
When viewing a ticket or project inside the Guides (How-To) page (the "pending ticket" and "pending project" views), the Sources section and Linked Entities section are still rendered directly in the main content body. This was already fixed on the Tickets page but the same cleanup was never applied to KB2HowtoPage.tsx.

Affected code in KB2HowtoPage.tsx:
- Pending ticket view (around lines 830-849): renders a "Sources" section with source cards inline.
- Pending ticket view (around lines 860-866): renders a "Linked Entities" section with badges inline.
- Pending project view (around lines 940-952): renders a "Sources" section with source ref cards inline.

These should only appear in the right panel, not in the main content area.

**Fix**: Remove the inline Sources and Linked Entities sections from both the pending ticket view and the pending project view in KB2HowtoPage.tsx. The `rightPanelSources` variable (lines 467-481) already collects these sources and sends them to `KB2RightPanel`. The right panel is already receiving the data — the inline sections are redundant duplication.
**Rerun**: None — UI-only.
