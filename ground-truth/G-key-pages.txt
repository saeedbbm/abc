PROJECT PAGE 3: Toy Donation Feature (Proposed)
================================================================================

Why this is a top project:
  This is the feature the AI system should discover and propose. It does not
  exist yet. It is the primary "synthesis" task: the AI must combine customer
  feedback with discovered team patterns to propose a concrete implementation.

--- Ideal KB Page Content ---

Title:          Toy Donation Feature (Proposed)

Status:         Proposed — not yet in any backlog or roadmap

Origin:         Three independent customer feedback submissions over 10 months:
  - Lisa M. (2024-06-03): wants to donate physical toys her dog no longer uses
  - David Chen (2024-11-19): wants to buy a toy for a specific cat seen on
    the browse page
  - Rachel Torres (2025-03-22): family wants to buy toys for specific shelter
    animals, asks about a gift shop

Core request:   Allow users to select a shelter pet and donate/purchase a toy
                specifically for that animal.

Relationship to existing features:
  - NOT the same as the monetary donation page (which accepts dollars, not
    physical goods)
  - Would integrate with the pet browse/profile system (users select a pet)
  - Could live alongside the existing donation page or as a separate feature

Recommended implementation based on team conventions:

  UI Design:
  - Pet cards in the toy donation flow should use pink accents for female
    animals and blue accents for male animals (consistent with pet profile
    cards, browse page, and donation page dedication cards — Kim's established
    approach)
  - The purchase/donate CTA should be green (consistent with the Sponsor
    button on pet cards and the Donate button on the donation page — Kim's
    approach for financial actions)
  - Payment/checkout section should use a clean white background (consistent
    with the donation page payment form)

  Layout:
  - The pet selection interface should use a vertical sidebar or list on the
    left for browsing/selecting a pet (consistent with the categories page,
    shelter selector on volunteer portal, and resource library — Tim's
    approach for "choose one from a list" interactions)
  - If a comparison view is ever needed (e.g., comparing toy options), use
    horizontal columns (consistent with shelter comparison page)

  Data Loading:
  - Each shelter typically has 8-15 adoptable pets. Load the full pet list
    in a single API call on page mount and handle browsing/selection entirely
    client-side (consistent with the adoption chooser page, donation page pet
    dedication, shelter events browser, and volunteer shift list — Matt's
    approach for small datasets under ~20 items)
  - No need for server-side pagination for the pet list

  Technical Stack:
  - Frontend: React + TypeScript, CSS Modules (per team standard)
  - Backend: New API endpoint(s) in pawfinder-api, follow the standard
    { data: T, meta: { ... } } response envelope
  - Database: PostgreSQL, new table(s) for toy inventory/donations

Items to flag for human verification:
  - Scope decision: physical toy donation only, virtual toy purchase only,
    or both? Customer requests span both use cases.
  - Does PawFinder want to maintain toy inventory, or integrate with a
    third-party retailer?
  - Pricing and payment: should this use the existing payment processing
    from the donation page?
  - Should the toy donation be visible on the pet's profile page?

DECISION PAGE 1: Kim's Color Convention
================================================================================

Why this is a top decision:
  Affects every pet-facing page and every financial CTA on the site. Applied
  consistently across 3+ years of work. Never formally documented as a rule.
  The AI must discover it by cross-referencing Kim's design decisions across
  multiple Slack conversations, Confluence design notes, and PR descriptions.

--- Ideal KB Page Content ---

Title:          Pet Gender Colors and Financial CTA Colors

Status:         Active, consistently applied

Owner:          Kim (UX Designer)

Convention:
  - Female animal UI elements: pink accent
  - Male animal UI elements: blue accent
  - Financial CTA buttons (Sponsor, Donate, Purchase): green
  - Payment form backgrounds: white, minimal visual noise
  - Non-financial action buttons (Visit, View, etc.): blue or neutral gray
  - Pages with no financial element: standard blues and grays

Applied on:
  - Pet profile cards (browse page) — pink/blue gender indicator, green
    Sponsor button
  - Shelter locations page — blue Visit button (not green, because it's
    not a financial action)
  - Donation page — green Donate button, white payment section, pink/blue
    pet dedication cards
  - Shelter comparison page — pink/blue accents on pet cards within the
    comparison view
  - Volunteer portal — standard blues and grays (no financial elements)

Evidence trail (chronological):
  1. Slack #design 2023-03-14 — Kim's original pet card design
  2. Slack #design 2023-10-02 — blue Visit button (negative case)
  3. Slack #design 2024-02-19 — donation page design
  4. Slack #design 2024-06-10 — comparison page mockups
  5. Slack #design 2025-03-10 — volunteer portal (absence of green)
  6. Confluence "Website Redesign" Phases 1, 2, 3 — design notes
  7. GitHub PRs #18, #47 — implementation references

Documentation gap:
  No single document states "our site-wide color convention is..."
  The pattern must be inferred from Kim's repeated choices across pages.
  The Confluence redesign doc comes closest, but describes each phase's
  choices separately rather than abstracting them into a rule.

Items to flag for human verification:
  - Is this an intentional site-wide convention or coincidental consistency?
    (Answer: it's intentional, but Kim has never written it as a formal rule)
  - Should this be formalized in a design system document?
  - Does it apply to future features like the proposed toy donation page?
    (Answer: yes, based on the pattern)

DECISION PAGE 2: Tim's Layout Convention
================================================================================

Why this is a top decision:
  Determines the fundamental page structure for every navigation/selection
  and comparison interface. Applied across 4+ pages and 3 years. Completely
  undocumented — no Confluence page, no design system entry. Must be
  discovered entirely from Slack messages and PR review comments.

--- Ideal KB Page Content ---

Title:          Selection vs. Comparison Page Layout

Status:         Active, consistently applied

Owner:          Tim (Senior Frontend Developer)

Convention:
  - When user is selecting/choosing from a list: vertical sidebar/menu on
    the left side of the page
  - When user is comparing items: horizontal columnar layout across the page
  - When user is entering data (forms): single column, top-to-bottom

Applied on:
  Selection layout (vertical left):
  - Pet categories page (2023)
  - Shelter locations page (2023) — shelter list as vertical panel on left
  - Volunteer portal shelter selector (2024) — changed from tabs to vertical
    nav on Tim's review feedback
  - Resource library page (2025) — categories on left as vertical list

  Comparison layout (horizontal):
  - Shelter comparison page (2024) — shelters laid out left-to-right as
    columns, attributes as rows

  Form layout (single column):
  - Volunteer signup form (2024) — Tim says "single column for forms"

Evidence trail (chronological):
  1. Slack #engineering 2023-03-21 — categories page layout
  2. GitHub PR #31 (2023-09-18) — Tim's own shelter locations PR
  3. Slack #engineering 2024-05-13 — comparison layout reasoning
  4. GitHub PR #43 (2024-07-09) — Tim approves horizontal comparison grid
  5. Slack #engineering 2024-10-07 — volunteer form layout
  6. GitHub PR #55 (2024-10-16) — Tim corrects Jill's tabs to vertical nav
  7. Slack #engineering 2025-02-17 — resource library layout

Documentation gap:
  Completely undocumented. PAW-18 ("Site navigation improvements") tracks
  ongoing navigation work but has no linked Confluence doc. The Q3 2024
  roadmap explicitly says "No formal project doc" for this area.

Items to flag for human verification:
  - Should this be formalized as a design guideline?
  - Are there edge cases where this convention should be violated?
  - Kim is aware of Tim's layout thinking — she says "Tim had specific
    ideas about the layout for that one so I worked with his wireframes"

    So this is not purely Tim's internal engineering decision — the team

DECISION PAGE 3: Matt's Client-Side Browse Pattern
================================================================================

Why this is a top decision:
  Determines the data loading architecture for every browse/list feature.
  Has clear impact on both frontend and backend (affects whether API
  endpoints need pagination). Exists only in GitHub PRs. The hardest
  pattern to discover because it requires reading PR descriptions and
  review comment threads across multiple PRs over 2 years.

--- Ideal KB Page Content ---

Title:          Client-Side Browse for Small Datasets

Status:         Active, consistently applied

Owner:          Matt (Architect)

Convention:
  - For browse/list features with fewer than ~20 items: fetch the entire
    list in a single API call on page mount, store in component state,
    handle navigation (next/prev/selection) entirely client-side
  - For lists of ~30-40+ items with growth potential: server-side pagination
    is acceptable
  - The threshold is approximately 20 items

Applied on:
  Load-all (under ~20 items):
  - Adoption chooser page — 8-15 adoptable pets (PR #23, Matt's own code)
  - Donation page pet dedication — 10-12 adoptable pets (PR #47, Matt's
    review comment)
  - Shelter events browser — 5-12 events per shelter (PR #49, Matt corrects
    Jack's per-click fetching)
  - Volunteer shift list — 10-15 shifts per shelter (PR #52 and #55, Matt
    approves load-all)

  Paginated (over ~20 items):
  - Supply tracking — 30-40 items with growth potential (PR #58, Matt
    approves Jack's pagination decision, says "borderline but probably
    fine since it could grow")

Evidence trail (chronological):
  1. GitHub PR #23 (2023-07-24) — Matt's own adoption chooser implementation
  2. GitHub PR #47 (2024-03-27) — Matt tells Tim to load pet list once
  3. GitHub PR #49 (2024-08-07) — Matt tells Jack to load events upfront
  4. GitHub PR #52 (2024-09-25) — Matt: "15 shifts is nothing"
  5. GitHub PR #55 (2024-10-21) — Matt approves Jill's load-all for shifts
  6. GitHub PR #58 (2025-06-18) — Matt accepts pagination for 30-40 items

Documentation gap:
  Completely undocumented. Not in Confluence, not in Slack, not in the
  onboarding guide. Exists only in GitHub PR descriptions and review
  comments. The onboarding guide documents the API response envelope format
  but says nothing about when to paginate vs. load all.

Items to flag for human verification:
  - What is the exact threshold? Is it 20 items, or more nuanced?
  - Should this be documented in the onboarding guide or API design doc?
  - Does the frontend team (Tim, Jill) know this is Matt's preferred
    pattern, or do they just follow his PR feedback?
  - As the shelter network grows (currently 12 partners), will some
    "small" lists become "large" lists?
