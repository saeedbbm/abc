DECISION 1: Kim's Color Convention
================================================================================

Canonical name:     Gender-Color and Money-Color UI Convention
Summary:            Pet-related UI elements use pink accents for female animals
                    and blue accents for male animals. Money-related UI elements
                    (donate buttons, sponsor buttons, payment forms) use green
                    for CTAs and white/neutral for backgrounds. Non-financial
                    action buttons use blue.
Made by:            Kim (UX designer)
Applies to:         All pet-facing UI: pet profile cards, browse page, donation
                    page, shelter comparison page, any page showing pet cards.
                    All financial CTAs across the site.
Status:             decided
Explicitly documented?  PARTIALLY. The specific color choices are mentioned in
                    the Confluence "Website Redesign" doc's design notes, but
                    they are described as choices Kim made for specific pages,
                    not as a site-wide rule. No document says "our convention
                    is pink/blue for gender and green for money."

All sources (chronological):

  1. Slack #design 2023-03-14
     Kim designing pet profile cards. Says "pink accent for the girls and blue
     for the boys" for gender indicator. Says "Sponsor Me button...going with
     green for that since it's money-related." Sarah agrees.

  2. Slack #design 2023-10-02
     Kim on shelter locations page. "Visit" button is "neutral blue...it's not
     a money action so doesn't need to be green." This establishes the
     negative case: non-financial buttons are NOT green.

  3. Slack #design 2024-02-19
     Kim on donation page. "main donate button I'm making green — it's a
     financial action." Payment section "white and clean." Pet dedication
     card "pink or blue accent depending on if it's a girl or boy animal."

  4. Slack #design 2024-06-10
     Kim on comparison page mockups. "animal cards on the comparison view
     still use the pink/blue accents."

  5. Slack #design 2025-03-10
     Kim on volunteer portal. "no financial stuff on this page so it's just
     our standard blues and grays." Reinforces the pattern by absence:
     no money = no green.

  6. Confluence "PawFinder Website Redesign" Phase 1
     "pink for female animals, blue for males" on gender indicator.
     "Sponsor Me button on each card is green since it's a financial action.
     Kim suggested green for any money-related call-to-action."

  7. Confluence "PawFinder Website Redesign" Phase 2
     "Visit CTA button on shelter cards is blue (not green, since it's not
     a financial action)."

  8. Confluence "PawFinder Website Redesign" Phase 3

     "pet card...uses the same pink/blue gender accent as the browse page."

Pattern elements:
  - Pink accent = female animal
  - Blue accent = male animal
  - Green = financial/money CTA (Sponsor, Donate)
  - White/neutral = payment/financial form backgrounds
  - Blue = non-financial action buttons (Visit)
  - Standard blues/grays = pages with no financial element

DECISION 2: Tim's Layout Convention
================================================================================

Canonical name:     Selection vs. Comparison Layout Convention
Summary:            When the user is choosing/selecting from a list (picking
                    one item to drill into), use a vertical menu/sidebar on
                    the left. When the user is comparing items side by side,
                    use a horizontal columnar layout across the page.
Made by:            Tim (Senior frontend developer)
Applies to:         All page layouts involving navigation or item presentation
                    on pawfinder/pawfinder-web. Applied to: pet categories page,
                    shelter comparison page, volunteer shift browser, resource
                    library page.
Status:             decided
Explicitly documented?  NO. Tim's layout thinking exists only in Slack messages
                    and GitHub PR review comments. There is no Confluence doc
                    that states this as a convention. PAW-18 ("Site navigation
                    improvements") tracks the work but has no linked doc.
                    The Q3 2024 roadmap mentions it has "No formal project doc."

All sources (chronological):

  1. Slack #engineering 2023-03-21
     Tim on pet categories page: "putting the category list as a vertical
     menu on the left side...feels right for a 'pick one of these' type
     interaction. the user is choosing, not comparing, so vertical nav
     makes more sense to me than tabs across the top."

  2. Slack #engineering 2024-05-13
     Tim on shelter comparison: "horizontal layout...shelters laid out left
     to right across the top so you can scan across and compare attributes.
     when you're comparing things, your eyes want to go left-right to see
     differences. it's different from when you're just picking something
     from a list."

  3. Slack #engineering 2024-10-07
     Tim on volunteer signup form: "single column for forms...forms are
     different from browse/comparison layouts." (Tangentially related —
     shows Tim thinks about layout categorically.)

  4. Slack #engineering 2025-02-17
     Tim on resource library page: "put the categories on the left as a
     vertical list, not tabs on top. when someone is choosing which section
     to look at, left side nav is better. tabs across the top work better
     when you're comparing stuff side by side."

  5. GitHub PR #43 — Tim's review comment (2024-07-09)
     Tim on Jill's shelter comparison PR: "The comparison grid looks right —
     horizontal columns for the shelters, rows for the attributes. That's
     the right call for a comparison view."

  6. GitHub PR #55 — Tim's review comment (2024-10-16)
     Tim on Jill's volunteer portal frontend: "I'd rethink the layout.
     Right now you have the shelter selector as tabs across the top, but
     the user is choosing which shelter to look at, not comparing shelters.
     Put the shelter list on the left as a vertical nav." Jill changes it.

  7. GitHub PR #31 — Tim's own PR (2023-09-18)
     Shelter locations page description: "The shelter list is a vertical
     scrollable panel on the left side." (The user is selecting a shelter,
     not comparing.)

Pattern elements:
  - Vertical left sidebar/menu = user is selecting/choosing/navigating
  - Horizontal columns/layout = user is comparing items
  - Single column top-to-bottom = forms (data entry)

DECISION 3: Matt's Client-Side Browse Pattern
================================================================================

Canonical name:     Client-Side Browse for Small Datasets
Summary:            For browse-and-choose flows where the dataset is small
                    (under ~20 items), load the entire list in a single API
                    call on page mount and handle next/prev or list navigation
                    entirely client-side. No server round-trip per click.
                    For larger or growing lists (30-40+ items), pagination
                    is acceptable.
Made by:            Matt (Architect)
Applies to:         All browse/list features on pawfinder/pawfinder-web and
                    corresponding pawfinder/pawfinder-api endpoints. Applied
                    to: adoption chooser (8-15 pets), donation page pet
                    dedication (10-12 pets), shelter events browser (5-12
                    events), volunteer shifts (10-15 shifts).
Status:             decided
Explicitly documented?  NO. Matt never writes this down as a rule. His pattern
                    exists primarily in his own PR code (PR #23) and his review
                    comments on others' PRs. The load-all-under-20 pattern
                    itself is not stated in Slack or Confluence, but the
                    boundary case (where Matt recommends pagination for a
                    larger ~30-40 item list) does appear in Slack 2025-06-09.

All sources (chronological):

  1. GitHub PR #23 — Matt's own PR (2023-07-24)
     Adoption chooser page. Description: "The page fetches the full list of
     adoptable pets from GET /api/v1/pets/adoptable on mount and stores them
     in component state. Navigation between pets is handled entirely
     client-side by indexing into the local array. No API call on prev/next —
     we typically have 8-15 adoptable pets at any given time so there's no
     reason to round-trip to the server for each click."

  2. GitHub PR #47 — Matt's review on Tim's PR (2024-03-27)
     Donation page. Matt's comment: "you're fetching the pet list fresh every
     time the user toggles it on. Just load them once when the page mounts
     and keep them in state. There are only like 10-12 adoptable pets at
     any time."

  3. GitHub PR #49 — Matt's review on Jack's PR (2024-08-07)
     Shelter events browser. Jack originally calls API on every Next/Prev
     click. Matt: "How many events does a shelter typically have?" Jack:
     "5-8, maybe 12 max." Matt: "just load them all upfront in one call
     when the page mounts. Store the array in state and navigate through
     it locally with the prev/next index. No need to make a server request
     every time the user clicks an arrow for a list that small."

  4. GitHub PR #52 — Matt's review on Jack's PR (2024-09-25)
     Volunteer portal backend. Matt: "The shift list for a shelter — you're
     right to return them all, 15 shifts is nothing."

  5. GitHub PR #55 — Matt's review on Jill's PR (2024-10-21)
     Volunteer portal frontend. Matt: "I see you're loading all shifts for
     the selected shelter at once — that's the right call, no need to
     paginate 10-15 items."

  6. GitHub PR #58 — Matt's review on Jack's PR (2025-06-18)
     Supply tracking backend. 30-40 supply items. Jack uses pagination.
     Matt: "for 30-40 items that's borderline but probably fine since it
     could grow." Approves pagination. THIS IS THE BOUNDARY CASE — Matt
     does NOT push for load-all when the list is larger.

Pattern elements:
  - Under ~20 items: load all in one API call, client-side navigation
  - ~30-40 items with growth potential: pagination is acceptable
  - The threshold is roughly 20 items
  - Rationale (implicit): UX feels snappier, fewer network requests
  - Backend implication: endpoints for small lists don't need pagination
    parameters, just return the full array
