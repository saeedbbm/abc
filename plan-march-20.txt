# KB2 Pipeline Fix Plan — March 20, 2026

This plan has two parts:
  Part 1: General Implementation Plan — generic pipeline fixes, no dataset-specific logic.
  Part 2: PawFinder Evaluation Appendix — dataset-specific expectations for offline evaluation.

Nothing in Part 1 should reference specific entity names, exact counts, specific people,
or ground truth. Part 2 is the only place those belong.

## Philosophy

Steps 1–11 must produce the truth.
Steps 12–18 only package, retrieve, narrate, guide, and verify that truth.

No downstream step should compensate for bad canonical names, wrong statuses,
wrong attribution, split conventions, or missing graph edges. Those must be
fixed upstream.

## Phase 1: Fix the Core Loop (Steps 5 + 8 + 9 + 10 + 11)

These five steps are interdependent:
  - Step 5 determines how many entities exist and their canonical names
  - Step 8 determines discovery nodes and proposed features
  - Step 9 fills/corrects attributes (status, decided_by) on the entity set
  - Step 10 groups decisions into conventions using decided_by from Step 9
  - Step 11 wires conventions into the graph

Start with Steps 9-11 on the current entity set to prove the convention
pipeline works, then loop back to fix Steps 5 and 8, then rerun 9-11 again.

### 1a. Step 9: Attribute Completion — make it corrective, not just fill-missing
File: `src/application/workers/kb2/pass1/attribute-completion.ts`

WHAT THIS STEP DOES:
  Step 9 takes all entity nodes from Step 5 (resolved entities) and Step 8
  (discovery nodes), clones them into a new execution_id, and enriches their
  attributes. It should fill missing values AND correct wrong values for:
  status, description, documentation_level, decided_by, rationale, scope.
  After this step, every entity should have complete, accurate public attributes
  that downstream steps can trust.

BUG 1 — Status filter only passes entities with MISSING status (never corrects wrong values).

  File: `attribute-completion.ts`, lines 212–217
  Current code:
    ```
    const needsStatus = allNodes.filter((n) => {
      const attrs = (n.attributes ?? {}) as Record<string, any>;
      if (n.type === "project") return !attrs.status || !VALID_PROJECT_STATUS.has(attrs.status);
      if (n.type === "process") return !attrs.status || !VALID_PROCESS_STATUS.has(attrs.status);
      return false;
    });
    ```
  Problem: `!attrs.status` is false when status exists (even if wrong). The
  `VALID_PROJECT_STATUS.has(attrs.status)` check only catches invalid ENUM values
  (like "planned"), not semantically wrong values (a project marked "proposed"
  whose linked Jira tickets are all Done). Entities with valid-but-wrong status
  are never sent to the LLM.

  CHANGE TO: A deterministic-first, LLM-second approach.

  Before the LLM batch loop (line 222), build a Jira status map from input
  snapshot docs and derive status deterministically where possible:
    ```
    // Deterministic status from Jira evidence (no LLM needed)
    const jiraStatusMap = buildJiraStatusMap(docs);  // ticket_key → Done/InProgress/Backlog
    for (const node of allNodes.filter(n => n.type === "project" || n.type === "process")) {
      const linkedTickets = findLinkedTickets(node, jiraStatusMap);
      if (linkedTickets.length > 0) {
        const allDone = linkedTickets.every(t => t.status === "Done");
        const anyInProgress = linkedTickets.some(t => t.status === "In Progress");
        const deterministic = allDone ? "completed" : anyInProgress ? "active" : null;
        if (deterministic) {
          // Set status directly, no LLM needed
          bulkOps.push({ updateOne: {
            filter: { node_id: node.node_id, execution_id: ctx.executionId },
            update: { $set: {
              "attributes.status": deterministic,
              "attributes._status_source": "jira-deterministic",
              "attributes._status_reasoning": `Linked tickets: ${linkedTickets.map(t => `${t.key} (${t.status})`).join(", ")}`,
            }},
          }});
          continue;  // skip LLM for this entity
        }
      }
      // Jira evidence is ambiguous, conflicting, or absent — send to LLM
      needsLLMStatus.push(node);
    }
    ```
  Then change the LLM filter to only include entities where Jira evidence
  was insufficient:
    ```
    const needsStatus = needsLLMStatus;  // only ambiguous/missing cases
    ```
  In the LLM prompt, still include Jira metadata as context for ambiguous cases,
  but let the deterministic path handle clear-cut cases without LLM cost.

  Also change the system prompt at line 234 to include:
    "When structured ticket status data is provided, give it priority over
    ambiguous conversational mentions. If evidence conflicts (e.g., some
    tickets Done but conversation mentions ongoing work), explain the
    conflict in your reasoning."

BUG 2 — decided_by filter only passes entities with MISSING decided_by.

  File: `attribute-completion.ts`, lines 286–289
  Current code:
    ```
    const needsDecisionFix = decisions.filter((n) => {
      const attrs = (n.attributes ?? {}) as Record<string, any>;
      return !attrs.decided_by || !attrs.rationale || !attrs.scope;
    });
    ```
  Problem: If Step 4 backfilled a wrong decided_by (its heuristic assigns
  "first team member mentioned in excerpts"), Step 9 sees a non-null value
  and skips the entity. The wrong attribution propagates to Step 10.

  This is the single most important fix in the plan. The core semantic
  change is: Step 9 must be CORRECTION-CAPABLE, not fill-only.
  - status must be allowed to be corrected, not just filled
  - decided_by must be allowed to be corrected, not just filled
  - an existing non-null value does NOT mean "correct"

  CHANGE TO:
    ```
    const needsDecisionFix = decisions;  // always re-evaluate all decisions
    ```
  Send ALL decisions to the LLM for decided_by inference, not just those
  with missing values. The prompt should include the current value so the
  LLM can explicitly confirm or correct it.

BUG 3 — decided_by prompt lacks provenance-based attribution rules.

  File: `attribute-completion.ts`, line 311
  Current prompt:
    ```
    const prompt = `For each decision, infer missing attributes from source excerpts.\n- decided_by: Who MADE this decision? Not who mentioned or reviewed it.\n...`;
    ```
  Problem: The prompt treats all source text equally. It doesn't distinguish
  who authored a PR vs who left a review comment vs who spoke in Slack. When
  a PR author writes code implementing a reviewer's design feedback, the
  prompt attributes the design decision to the PR author, not the reviewer.

  CHANGE TO:
    1. Add structured provenance metadata to the prompt context. For each
       decision, parse its source_refs to extract:
       - PR source_refs: include `PR Author: X, PR Reviewers: [Y, Z]` explicitly
       - Slack source_refs: include `Slack Speaker: X` explicitly
       - Confluence source_refs: include `Confluence Author: X` explicitly
       These ROLES are the key to correct attribution.

    2. Add explicit provenance rules to the system prompt (line 311):
       ```
       - IMPLEMENTER and DECISION-MAKER are SEPARATE roles.
       - A PR AUTHOR who implements code is the IMPLEMENTER unless they also
         originated the design.
       - A PR REVIEWER who gives architectural guidance, design feedback, or
         pattern direction in review comments is the DECISION-MAKER for
         decisions stated in those comments.
       - A DESIGNER/ARCHITECT who says "I want this to use X pattern" in
         Slack or a review comment is the DECISION-MAKER, not the person
         who implements it in code.
       - When a PR author writes code that follows a reviewer's design
         direction, the reviewer is the decision-maker.
       - If the current decided_by appears to be the implementer rather
         than the person who originated the architectural choice, CORRECT it.
       ```

    3. Include the existing decided_by value in the prompt for each entity:
       `Current decided_by: "Tim" (from earlier pipeline step — may be wrong)`
       so the LLM can explicitly confirm or correct.

    4. Structure the prompt so provenance is first-class, not buried in text:
       ```
       Decision: "Client-Side Pet Navigation"
       Current decided_by: "Tim"
       Provenance:
         - PR #47: Author=Tim, Reviewers=[Matt]
         - PR #47 review comment by Matt: "let's load all pets on mount and
           navigate client-side, the dataset is small enough"
         - Slack #dev 2024-03-14: Tim says "implementing Matt's suggestion for
           client-side browsing"
       Question: Who MADE this architectural decision?
       ```

BUG 4 — Uses getCrossCheckModel (GPT-4o) for decided_by inference.

  File: `attribute-completion.ts`, line 293
  Current: `const model = getCrossCheckModel(ctx.config?.pipeline_settings?.models);`
  Problem: decided_by attribution is the most critical inference in the
  pipeline (it determines convention ownership in Step 10). GPT-4o is fast
  but not the best at multi-source reasoning.

  CHANGE TO:
    Split the step into two model paths:
    - Status inference: keep getCrossCheckModel (GPT-4o) — status is
      largely deterministic from Jira data.
    - decided_by/rationale/scope inference: use getReasoningModel (Claude Opus).
    Add the import at the top of the file:
      `import { getReasoningModel, getReasoningModelName, ... } from "@/lib/ai-model";`
    At line 293, use reasoning model for the decision batch:
      `const model = getReasoningModel(ctx.config?.pipeline_settings?.models);`

BUG 5 — Per-type batching not implemented (all entity types in one batch).

  File: `attribute-completion.ts`, lines 222–281 (status) and 295–365 (decided_by)
  Current: One batch loop for all entities, mixed types.
  CHANGE: Separate into per-type batch loops:
    - Projects: batches of 10–12, with linked Jira ticket statuses as metadata
    - Decisions: batches of 6–8, with PR author/reviewer metadata
    - Processes: batches of 10
    - Team members: 1 batch (only ~6 in any dataset)

BUG 6 — Cross-check for high-impact decided_by not implemented.

  After the reasoning model fills decided_by, add a cross-check pass:
  - Select the N decisions with highest impact (those where `decided_by`
    appears on 3+ decisions — potential convention originators).
  - Send these to GPT-4o with the same evidence.
  - If the two models disagree, run a third adjudication call with both
    perspectives.
  - Log: which decisions were cross-checked, whether models agreed,
    adjudication results if any.

LOGGING REQUIREMENTS:
  The step artifact must include:
  - total_entities_processed
  - statuses_corrected (count where status was CHANGED)
  - statuses_filled (count where status was added from null)
  - statuses_confirmed (count where existing status was verified correct)
  - decided_by_corrected (changed from one value to another)
  - decided_by_filled (added from null)
  - decided_by_confirmed (existing value verified correct)
  - For each correction: { entity_name, field, old_value, new_value,
    reasoning, model_used }
  - Per-batch LLM call details: which entities, what model returned,
    which values accepted/rejected
  - A summary table of all decision entities: decided_by before vs after

MODELS:
  - Claude Opus (reasoning) for decided_by on decisions
  - GPT-4o (cross-check) for decided_by cross-validation on high-impact decisions
  - GPT-4o for status inference
  - GPT-4o for description generation (keep as-is)

RERUN: After fixing, rerun Steps 9 → 10 → 11 (shortest loop).
Then loop back with Step 5 + 8 fixes and rerun 5 → 8 → 9 → 10 → 11 (core loop).

### 1b. Step 10: Pattern Synthesis — per-cluster, multi-phase, attribution verification
File: `src/application/workers/kb2/pass1/pattern-synthesis.ts`

WHAT THIS STEP DOES:
  Step 10 analyzes all decision entities to discover CROSS-CUTTING CONVENTIONS —
  recurring patterns where the same person makes the same type of choice
  across multiple features over time. It should find hidden conventions,
  attribute each to the correct person, and create first-class convention nodes
  in the graph with complete evidence.

BUG 1 — Single LLM call for ALL decisions.

  File: `pattern-synthesis.ts`, lines 80–110
  Current code: ALL decisions are concatenated into one `decisionsText` string
  and sent to a single `structuredGenerate` call (line 102). One prompt with
  dozens of decisions means the LLM's attention is spread thin.

  CHANGE TO: Multi-phase approach:

  Phase 1 — Programmatic candidate clustering (before any LLM call).
    Add new function `clusterDecisionsBySimilarity(decisions)`:
    - Group decisions by their `decided_by` attribute.
    - Within each person's decisions, compute pairwise token similarity on
      `display_name + (attrs.description ?? "") + source excerpts`.
    - Cluster decisions with similarity > 0.4 using single-link clustering.
    - A cluster of 3+ similar decisions by the same person = candidate convention.
    Insert this before line 80. Log: per-person clusters with similarity scores.

  Phase 2 — Per-cluster evidence collection.
    For each candidate cluster:
    - Collect ALL source documents from ALL constituent decisions' source_refs.
    - Include FULL relevant sections from source docs (not truncated to 1500 chars
      as the current line 94 does: `d.content.slice(0, 1500)`).
    - Include structured metadata: PR author, PR reviewer, Slack speaker.
    Replace the single `docContext` construction (lines 92–95) with per-cluster
    evidence packs.

  Phase 3 — Per-cluster synthesis with Claude Opus.
    Replace the single `structuredGenerate` call (line 102) with one call
    per candidate cluster. The prompt for each cluster provides:
    - All constituent decisions with full source excerpts
    - Structured decided_by values from Step 9
    - Instruction: "If this pattern has multiple aspects (e.g., color for
      gender AND color for money), describe ALL aspects in the pattern_rule.
      Do not split into separate conventions."
    - Instruction: "The originator is the person with the strongest
      provenance-weighted evidence across constituent decisions. Do not
      simply count decided_by occurrences — weigh PR review comments
      and Slack design statements higher than PR authorship."
    - Documentation check: "If this pattern is fully described as a rule
      in a formal document (onboarding guide, spec, wiki), set
      documentation_level to 'documented'. If only described per-instance
      but never as a general rule, set to 'partially_documented'. If no
      documentation at all, set to 'undocumented'."

  Phase 4 — Attribution verification with GPT-4o.
    After synthesis, for each convention:
    - One GPT-4o call: "Based on the evidence, is {person} the correct
      originator? Look at PR authorship vs review comment authorship vs
      Slack speaker."
    - If GPT-4o disagrees: Claude Opus adjudicates with both perspectives.
    Add this as a new section after the convention node creation loop (after line 156).

Phase 5 — Convention consolidation (merge adjacent subclusters).
    After Phase 3 produces per-cluster conventions, check whether any pair
    of conventions from the same owner should be merged into one broader
    pattern. Merge when they:
    - have the same established_by
    - have overlapping source evidence (shared doc_ids)
    - represent complementary facets of one larger reusable rule (e.g.,
      "color for gender" + "color for money" = one color convention)
    This is a programmatic check followed by one LLM call for ambiguous cases:
    "These two conventions share the same owner and overlapping evidence.
    Are they facets of one broader convention? If yes, provide the merged
    convention_name, pattern_rule, and combined constituent_decisions."
    Log: which conventions were considered for merging, which were merged,
    which were kept separate.

BUG 2 — Source ref deduplication too aggressive.

  File: `pattern-synthesis.ts`, line 131
  Current: `.filter((r, i, arr) => arr.findIndex((x) => x.doc_id === r.doc_id) === i)`
  Problem: Deduplicates by doc_id alone. If a pattern appears in Phase 1,
  Phase 2, and Phase 3 sections of the same Confluence doc (same doc_id),
  only the first ref is kept.

  CHANGE TO:
    `.filter((r, i, arr) => arr.findIndex((x) => x.doc_id === r.doc_id && x.section_heading === r.section_heading) === i)`
  Deduplicate by doc_id + section_heading, preserving multiple sections from
  the same document.

BUG 3 — Documented standards are labeled as hidden conventions.

  File: `pattern-synthesis.ts`, lines 133–156 (convention node creation)
  Current: All conventions from the LLM are written with `is_convention: true`
  and `documentation_level: "undocumented"` (line 148).
  Problem: A pattern that is already described as a formal rule in a wiki/spec
  document is NOT a hidden convention. Creating a new convention node for it
  produces a duplicate of the existing decision entity.

  CHANGE TO:
  First, update the ConventionSchema (line 11) to include `documentation_level`:
    Add to the schema: `documentation_level: z.enum(["undocumented", "partially_documented", "documented"])`
  A convention is "partially_documented" when specific instances are documented
  (e.g., individual page color choices are in Confluence) but the general
  cross-cutting rule is never stated as such. This is distinct from "undocumented"
  (no documentation at all) and "documented" (the rule itself is written down).

  After the LLM returns conventions (line 118), add a filtering step:
    ```
    const filteredConventions = conventions.filter(conv => {
      // If a convention maps 1:1 to an existing documented decision, skip it
      if (conv.constituent_decisions.length === 1) {
        const existingDec = decisionsByName.get(conv.constituent_decisions[0].toLowerCase().trim());
        if (existingDec && (existingDec.attributes as any)?.documentation_level === "documented") {
          return false;  // already exists as a documented decision entity
        }
      }
      // Fully documented standards should not become hidden convention nodes
      if (conv.documentation_level === "documented") return false;
      return true;
    });
    ```
  Use `filteredConventions` instead of `conventions` for the node creation loop.
  When creating convention nodes (line 148), use the LLM's documentation_level
  instead of hard-coding "undocumented":
    `documentation_level: conv.documentation_level ?? "undocumented"`
  Log rejected candidates with reasons.

BUG 4 — Doc context truncated too aggressively.

  File: `pattern-synthesis.ts`, lines 92–95
  Current: `docs.slice(0, 20)` and `d.content.slice(0, 1500)`
  This is replaced by Phase 2 per-cluster evidence collection (see above).
  Remove the global `docContext` variable entirely.

LOGGING REQUIREMENTS:
  The step artifact must include:
  - Phase 1 results: candidate clusters per person, with decision names
    and similarity scores
  - Phase 3 results: for each convention, the full LLM output including
    convention_name, established_by, pattern_rule, constituent_decisions,
    documentation_level, confidence
  - Phase 4 results: cross-check agreement/disagreement, adjudication results
  - Final conventions: the complete list with all attributes
  - Rejected candidates: clusters or conventions filtered out, with reasons
    (e.g., "documented standard", "insufficient evidence")
  - Summary: "Found N hidden conventions. Documented standards filtered: M."

MODELS:
  - Claude Opus (reasoning) for per-cluster convention synthesis
  - GPT-4o (cross-check) for attribution verification
  - Claude Opus (reasoning) for adjudication when models disagree

RERUN: After fixing, rerun Steps 10 → 11 (or 9 → 10 → 11 if decided_by changed).

### 1c. Step 11: Graph Re-enrichment — fix edge quality and targeting
File: `src/application/workers/kb2/pass1/graph-re-enrichment.ts`

WHAT THIS STEP DOES:
  Step 11 wires the graph for conventions and discovery nodes. It creates:
  - CONTAINS edges from each convention to its constituent decisions
  - PROPOSED_BY edges from each convention to its establishing team member
  - APPLIES_TO edges from conventions to proposed features (via LLM matching)
  - RELATED_TO edges from discovery nodes to related existing entities
  After this step, the graph should be traversable:
  proposed feature → APPLIES_TO → convention → PROPOSED_BY → person

EXECUTION_ID WIRING NOTE:
  Previous analysis suspected Step 11 could not find Step 10's convention
  nodes due to an execution_id mismatch. This was a misdiagnosis — the zero-edge
  report came from the broken dump script (Phase 0). Convention edges ARE
  written to the DB. However, as a precaution, add debug logging to confirm
  the execution_id lookup works correctly on reruns.

BUG 1 — constituent_decisions name matching fails on type prefixes.

  File: `graph-re-enrichment.ts`, lines 99–101
  Current code:
    ```
    const constituents = (attrs.constituent_decisions ?? []) as string[];
    for (const decName of constituents) {
      const decNode = nodeByName.get(decName.toLowerCase().trim());
    ```
  Problem: Step 10's LLM returns constituent_decisions as strings like
  "Decision: Client-side pet navigation without per-click API calls".
  The `nodeByName` map is keyed by `node.display_name.toLowerCase().trim()`
  which does NOT include the "Decision:" prefix. The lookup silently fails.

  CHANGE TO:
    ```
    const constituents = (attrs.constituent_decisions ?? []) as string[];
    for (const rawDecName of constituents) {
      const decName = rawDecName
        .replace(/^decision:\s*/i, "")
        .replace(/^process:\s*/i, "")
        .replace(/^pr\s*#/i, "PR #");
      let decNode = nodeByName.get(decName.toLowerCase().trim());
      // Fallback: try token similarity matching
      if (!decNode) {
        for (const [key, node] of nodeByName) {
          if (node.type === "decision" && tokenSimilarity(decName, key) > 0.7) {
            decNode = node;
            break;
          }
        }
      }
    ```
  Import `tokenSimilarity` or add a local copy. Log which names matched
  via exact lookup vs fallback vs failed.

BUG 2 — APPLIES_TO filter too broad (matches every proposed entity).

  File: `graph-re-enrichment.ts`, lines 144–147
  Current code:
    ```
    const proposedFeatures = allNodes.filter((n) => {
      const cat = (n.attributes as any)?.discovery_category ?? "";
      return cat.startsWith("proposed_") || (n.attributes as any)?.status === "proposed";
    });
    ```
  Problem: `status === "proposed"` matches every entity with proposed status,
  including dozens of speculative noise tickets. The LLM then matches
  conventions to all of them, producing noisy APPLIES_TO edges instead of
  converging on the actual proposed features that matter.

  The goal is NOT to maximize APPLIES_TO count. The goal is to make the graph
  naturally support: proposed_feature → APPLIES_TO ← convention → PROPOSED_BY → owner,
  so that a single feature node can inherit all relevant conventions via traversal.

  CHANGE TO:
    ```
    const proposedFeatures = allNodes.filter((n) => {
      const cat = (n.attributes as any)?.discovery_category ?? "";
      return (
        cat === "proposed_from_feedback" ||
        cat === "proposed_project" ||
        (n.type === "project" && (n.attributes as any)?.status === "proposed")
      );
    });
    ```
  Only match conventions to real proposed features/projects, not generic
  speculative ticket noise. A "proposed_ticket" for a minor bug fix should
  NOT receive APPLIES_TO edges from design conventions.

BUG 3 — No edge confidence/provenance metadata.

  File: `graph-re-enrichment.ts`, lines 106–116 (CONTAINS), 124–137 (PROPOSED_BY),
  and 188–200 (APPLIES_TO).
  Current: Edges have `evidence` text but no `source_step` or `match_method`.

  CHANGE: Add to every new edge object:
    ```
    source_step: "step-11",
    match_method: "exact_name" | "prefix_stripped" | "token_similarity" | "llm",
    ```
  This allows debugging which edges came from which matching strategy.

LOGGING REQUIREMENTS:
  The step artifact must include:
  - execution_id_debug: { step9_exec_id, step10_exec_id, step10_node_count,
    step10_convention_count }
  - convention_wiring: for each convention, { name, contains_edges_created,
    contains_targets_missed (names that didn't resolve), proposed_by_created }
  - applies_to_results: for each APPLIES_TO edge, { convention, feature,
    relevance, confidence }
  - edge_summary: { total, by_type: { CONTAINS, PROPOSED_BY, APPLIES_TO,
    RELATED_TO } }

MODELS:
  - Claude Sonnet (fast) for APPLIES_TO matching (keep as-is)

RERUN: After fixing, rerun Step 11.

### 1d. Step 5: Entity Resolution — stronger evidence-based merging
File: `src/application/workers/kb2/pass1/entity-resolution.ts`

WHAT THIS STEP DOES:
  Step 5 takes the over-extracted entity set from Steps 3+4 and merges
  duplicates into canonical entities. This is THE MAIN merge/canonicalization
  step. Entity counts should be consistent with the input data's actual
  initiative structure and should not remain heavily fragmented after resolution.

  IMPORTANT: Step 5 merges entities of the SAME KIND. A project entity and
  a decision entity that describe related work should stay SEPARATE but LINKED.
  They represent different concepts (implementation work vs architectural choice).
  Step 5 should record cross-type relationships for Step 6 but NOT merge them.

BUG 0 — No deterministic identity anchors.

  Before any similarity-based candidate generation, build deterministic
  identity maps from stable source identifiers:
    ```
    // Canonicalize tickets by ticket key (e.g., "PAW-15" is one identity)
    const ticketByKey = new Map<string, KB2GraphNodeType>();
    for (const node of nodes) {
      if (node.type === "ticket" && /^[A-Z]{2,10}-\d+$/.test(node.display_name)) {
        ticketByKey.set(node.display_name.toUpperCase(), node);
      }
    }
    // Canonicalize PRs by repo + PR number (e.g., "pawfinder-web PR #47")
    const prByKey = new Map<string, KB2GraphNodeType>();
    for (const node of nodes) {
      if (node.type === "pull_request") {
        const prMatch = node.display_name.match(/(.+?)\s*PR\s*#(\d+)/i);
        if (prMatch) prByKey.set(`${prMatch[1].trim().toLowerCase()}#${prMatch[2]}`, node);
      }
    }
    ```
  When multiple nodes share the same ticket key or PR identity, merge them
  deterministically without LLM involvement. These are identity-based merges,
  not similarity-based.

BUG 1 — No evidence-based blocking signals beyond name similarity.

  File: `entity-resolution.ts`, lines 250–295 (candidate generation)
  Current: The blocking phase uses only name/alias similarity, substring matching,
  and cross-type name similarity. It does NOT use shared Jira tickets or PRs as
  first-class merge signals.

  CHANGE: After the existing same-type candidate generation loop (line 295),
  add a new evidence-based blocking pass. Shared Jira or PR evidence should be
  a strong merge SIGNAL, not an automatic merge rule — the LLM still decides:
    ```
    // Evidence-based blocking: shared Jira ticket keys
    const ticketToNodes = new Map<string, KB2GraphNodeType[]>();
    for (const node of nodes) {
      if (mergedNodeIds.has(node.node_id)) continue;
      const allText = [node.display_name, ...node.source_refs.map(r => r.title), ...node.source_refs.map(r => r.excerpt)].join(" ");
      for (const m of allText.matchAll(/\b([A-Z]{2,10}-\d+)\b/g)) {
        const ticket = m[1].toUpperCase();
        const list = ticketToNodes.get(ticket) ?? [];
        list.push(node);
        ticketToNodes.set(ticket, list);
      }
    }
    for (const [ticket, group] of ticketToNodes) {
      const unique = [...new Map(group.map(n => [n.node_id, n])).values()];
      for (let i = 0; i < unique.length; i++) {
        for (let j = i + 1; j < unique.length; j++) {
          if (unique[i].type !== unique[j].type) continue;
          const key = candidateKey(unique[i], unique[j]);
          if (existingPairKeys.has(key)) continue;
          existingPairKeys.add(key);
          candidates.push({ nodeA: unique[i], nodeB: unique[j],
            reason: `shared Jira ticket ${ticket}`, score: 0.95 });
        }
      }
    }
    // Same for shared PR numbers
    ```

## Phase 6: Judge System Integration

### Problem
The current LLM-as-Judge tab is too generic. It primarily looks at the step
artifact and a broad hardcoded prompt. This means it can say "pass" even when
the step clearly failed its actual pipeline goal. It cannot distinguish:
- a local step failure from an upstream step failure
- a step that filled attributes from one that corrected wrong attributes
- noisy ticket discoveries from valuable feature discoveries

### Solution: Step-Specific Goal Files
A new `goals/` directory provides the judge with step-specific contracts that
define what each step is responsible for, what success looks like, and what
the step must NOT do. These are general-purpose quality contracts, NOT
benchmark answer keys.

Files:
  - `goals/goal.txt` — overall product goal and end-to-end success criteria
  - `goals/README.md` — usage documentation
  - `goals/steps/pass1-step-01.yaml` through `goals/steps/pass1-step-18.yaml`

Each step YAML contains:
  - `purpose` — what the step is supposed to do
  - `inputs` — what it reads
  - `primary_outputs` — what it produces
  - `judge_should_look_at` — where to focus evaluation effort
  - `must_do` — non-negotiable requirements
  - `must_not_do` — anti-patterns the judge should flag
  - `success_signals` — what good output looks like
  - `failure_signals` — what bad output looks like
  - `deterministic_checks` — programmatic checks (counts, coverage %)
  - `go_no_go` — when to proceed vs block downstream steps

### 6a. Update the Judge to Read Goal Files

The judge API route (likely `app/api/[companySlug]/kb2/judge/route.ts`) must
be updated to:

  1. Load `goals/goal.txt` into the judge's system context.
  2. Load the matching `goals/steps/pass1-step-{stepNumber}.yaml` for
     whatever step is being judged.
  3. Inject both into the judge prompt BEFORE the artifact inspection.

  Code change: In the judge route, add at the top of the judge logic:
    ```
    import { readFileSync } from "fs";
    import { parse as parseYaml } from "yaml";
    import path from "path";

    const goalText = readFileSync(path.join(process.cwd(), "goals", "goal.txt"), "utf-8");
    const stepGoalPath = path.join(process.cwd(), "goals", "steps", `pass1-step-${String(stepNumber).padStart(2, "0")}.yaml`);
    const stepGoal = parseYaml(readFileSync(stepGoalPath, "utf-8"));
    ```

  Then include in the judge system prompt:
    ```
    You are evaluating Step ${stepNumber}: ${stepGoal.step_name}.

    OVERALL PIPELINE GOAL:
    ${goalText}

    THIS STEP'S CONTRACT:
    Purpose: ${stepGoal.purpose}
    Must do: ${stepGoal.must_do.join("\n")}
    Must not do: ${stepGoal.must_not_do.join("\n")}
    Success signals: ${stepGoal.success_signals.join("\n")}
    Failure signals: ${stepGoal.failure_signals.join("\n")}

    Judge this step ONLY on its own responsibility. Do not blame it for
    missing upstream truth unless its contract says it owns that correction.
    ```

  4. Run deterministic checks from the YAML programmatically (not via LLM):
    ```
    for (const check of stepGoal.deterministic_checks) {
      const actual = extractMetric(artifact, check.key);
      const passed = evaluateCheck(actual, check.mode, check.target);
      deterministicResults.push({ key: check.key, actual, target: check.target, passed });
    }
    ```
    Report deterministic check results separately from the LLM's qualitative
    assessment. This gives a "hard" score (pass/fail on measurables) plus a
    "soft" score (LLM's holistic judgment).

  5. When step artifacts include sampled entities/edges/pages, the judge should
     INSPECT those samples against the `judge_should_look_at` guidance from
     the step YAML. Currently the judge only reads the summary artifact — it
     should also sample real DB records when available.

### 6b. Benchmark Overlay (Optional, for PawFinder-specific evaluation)

For benchmark-aware judging, support an optional overlay layer:
  - `benchmarks/pawfinder/global.md` — PawFinder-specific expectations
  - `benchmarks/pawfinder/steps/pass1-step-XX.yaml` — per-step benchmark criteria

When benchmark mode is enabled, the judge should:
  1. Read the base goal + step YAML (general quality contract)
  2. Read the benchmark overlay for that step
  3. Read relevant `ground-truth/*.txt` files
  4. Report TWO scores:
     - General pipeline quality (from base contract)
     - Benchmark fit (from overlay + ground truth)
  These must be reported separately. A step can pass the general quality
  check while failing the benchmark fit (upstream entities are well-formed
  but happen to be wrong for this specific dataset).

The benchmark overlay files do NOT exist yet. They should be created from
the content currently in Part 2 of this plan. For now, the judge should
work with the base goal files only.

### 6c. Anti-Cheating Boundary

The goal files MUST NOT:
- contain benchmark-specific names, people, counts, or expected conventions
- cause downstream steps to "pretend success" when upstream truth is bad
- be fed into pipeline generation prompts as hidden answer keys
- replace benchmark evaluation with general quality assessment

The goal files SHOULD:
- make the judge more specific about what each step owns
- help the judge distinguish upstream failure from local failure
